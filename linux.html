Linux Basics
File system, navigation, file ops, permissions, users
What is Linux?
The OS under everything — servers, Android, Docker
▼
Simple: Linux is an open-source operating system kernel. Think of it as the engine of a car — the programs you run are the body. Linux runs on ~96% of web servers, all Android phones, and every Docker container.
Analogy: Windows is like a pre-built IKEA room. Linux is like getting the raw materials — you decide what goes where, but you have full control.
uname -a Show kernel version + system info
lsb_release -a Show Linux distribution name
whoami Show current logged-in user
hostname Show machine name
File System Structure
Where everything lives — understanding /
▼
Analogy: Linux has one giant tree rooted at / (called "root"). There is no C:\ or D:\ — everything hangs off this single root.
/ Root of everything
/home/you Your personal folder (like Desktop on Windows)
/etc All config files (nginx, ssh, etc.)
/var/log Log files — first place to debug issues
/usr/bin User programs (ls, grep, python...)
/tmp Temp files, wiped on reboot
/proc Virtual fs — live kernel/process info
ℹ Docker containers have their own isolated / tree — that's one of the key things that makes them work!

Create, copy, move, delete files and folders
▼

File Permissions
chmod, chown — who can read/write/execute
▼
Simple: Every file has 3 sets of permissions — for the owner, group, and everyone else. Each set has 3 flags: read, write, execute.
# ls -la shows permissions: -rwxr-xr-- 1 rahul dev 1024 script.sh # rwx = owner can read, write, execute # r-x = group can read and execute # r-- = others can only read
chmod +x file Make file executable
chmod 755 file owner=rwx, group=rx, others=rx
chmod 644 file owner=rw, everyone=read only
chown user:group file Change owner and group
Numbers: r=4, w=2, x=1. Add them up. 7=rwx, 5=rx, 4=r. So 755 means full owner, read+exec for others.
Users & sudo
root, sudo, user management
▼
Simple: Linux has a superuser called root who can do anything. Normal users have limited access. sudo lets you run a single command as root.
sudo command Run command as root
sudo su Switch to root shell (use with care!)
adduser username Create new user
passwd username Set/change user password
id Show current user's UID and groups
⚠ Docker containers often run as root by default — this is a security concern in production. We'll cover this in the Docker section.

Intermediate Linux
Processes, pipes, grep, networking, package management, shell scripting
Processes
ps, top, kill — managing running programs
▼
Simple: Every program running on Linux is a "process" with a unique PID (Process ID). The kernel manages all processes — you can inspect, pause, or kill them.
ps aux List all running processes
top Live process monitor (CPU, memory)
htop Better top — colorful, interactive
kill PID Send termination signal to process
kill -9 PID Force kill (SIGKILL) — immediate
pkill nginx Kill process by name
pgrep nginx Get PID of process by name
nohup cmd & Run in background, survives terminal close
ℹ Docker containers are essentially just processes on the host OS, managed by the Docker daemon. ps aux on the host shows all container processes.
Pipes & Redirection
The most powerful Linux concept — chaining commands
▼
Simple: The pipe | sends the output of one command as input to another. Redirection > saves output to a file. This is how Linux does complex tasks with simple commands.
Analogy: Imagine a water pipe. Each command is a filter — water flows through them in sequence, transformed at each step.
cmd1 | cmd2 Send output of cmd1 into cmd2
cmd > file Save output to file (overwrite)
cmd >> file Append output to file
cmd 2> err.log Redirect errors to file
cmd > /dev/null Throw output away (silence)
# Find all running nginx processes $ ps aux | grep nginx # Count how many files in /etc $ ls /etc | wc -l 186 # Save all errors from a script to log $ ./deploy.sh 2> error.log

Search and manipulate text like a pro
▼
grep — search text

Environment Variables
export, .env, $PATH — critical for Docker
▼
Simple: Env vars are key=value pairs available to all programs. They configure apps without changing code. Docker -e flag and .env files use this exact system.
export MY_VAR=hello Set env var for current session
echo $MY_VAR Print value of variable
env List all environment variables
echo $PATH Show where Linux looks for programs
printenv DB_URL Print specific env var
ℹ In Docker: docker run -e DB_URL=postgres://... app sets env vars inside the container — same concept!

Systemd & Services
Managing background services — nginx, mysql, docker daemon
▼
Simple: Systemd is Linux's init system — it starts all services when the machine boots. You use systemctl to manage these services.
systemctl status nginx Check if nginx is running
systemctl start nginx Start nginx service
systemctl stop nginx Stop nginx
systemctl restart nginx Stop and start again
systemctl enable nginx Auto-start on boot
systemctl disable nginx Don't auto-start on boot
journalctl -u nginx -f Follow logs for a specific service
ℹ systemctl enable docker makes the Docker daemon start on boot — important for servers.
Cron Jobs
Schedule tasks to run automatically
▼
Simple: Cron is Linux's task scheduler. You write a "crontab" entry with a time pattern and a command. Linux runs it automatically at that time.
# crontab format: # MIN HOUR DAY MONTH WEEKDAY command # (0-59)(0-23)(1-31)(1-12)(0-6=sun-sat) */5 * * * * /scripts/backup.sh # every 5 min 0 2 * * * /scripts/cleanup.sh # daily at 2am 0 0 * * 1 /scripts/report.sh # every Monday midnight
crontab -e Edit your cron jobs
crontab -l List all current cron jobs

df -h Disk space of all partitions (human readable)

Namespaces & cgroups
The kernel features that power Docker containers
▼
Simple: This is how Docker actually works at the kernel level. You don't need to use these directly, but understanding them makes Docker intuitive.
Namespaces = Isolation. Each container gets its own view of: process IDs (pid), network interfaces (net), filesystem (mnt), hostname (uts). Processes inside can't see outside.

cgroups = Resource limits. They control how much CPU, RAM, and disk I/O a container can use. docker run --memory=512m uses cgroups under the hood.
# See namespaces of a running process $ ls -la /proc/1234/ns/ # Docker uses these automatically # docker run = new PID + net + mnt namespace

SSH Keys & Tunneling
Passwordless login, port forwarding
▼
# Generate SSH key pair $ ssh-keygen -t ed25519 -C "your@email.com" # Creates: ~/.ssh/id_ed25519 (private) + id_ed25519.pub (public) # Copy public key to remote server $ ssh-copy-id user@server # SSH tunnel: access remote port 5432 locally $ ssh -L 5432:localhost:5432 user@server
ℹ SSH keys are used everywhere in DevOps — GitHub authentication, server access, CI/CD pipelines, and Docker image registry access.
Firewall — ufw & iptables
Control network access in/out of the server
▼
ufw status Check firewall status
ufw allow 80 Allow HTTP traffic
ufw allow 22 Allow SSH (do this before enabling!)
ufw deny 3306 Block MySQL from outside
ufw enable Turn on the firewall
⚠ Docker bypasses ufw and writes directly to iptables! If you expose a Docker port with -p, it's publicly accessible even if ufw blocks it. Use Docker networks to limit exposure.

Volumes & File Permissions
The #1 Docker headache — solved with Linux knowledge
▼
Common problem: You mount a volume into a container and get "permission denied." This happens because the container's user UID doesn't match the host file's owner UID.
# Check who owns files on host $ ls -la ./data drwxr-xr-x 1000 1000 data/ # owned by UID 1000 # Check UID inside container $ docker exec myapp id uid=0(root) # running as root = no permission issue # Fix: make your user own the directory $ chown -R 1000:1000 ./data
⚠ Always think about file ownership when mounting Docker volumes. chown and chmod are your solutions.

Ports, interfaces, host networking

Log Debugging with Linux Tools
grep + tail + docker logs = powerful debugging
▼
# Follow container logs (like tail -f) $ docker logs -f myapp # Pipe docker logs into grep — find errors only $ docker logs myapp 2>&1 | grep -i "error" # Last 100 lines $ docker logs --tail 100 myapp # Show logs since 30 minutes ago $ docker logs --since 30m myapp
ℹ All your Linux knowledge (pipe, grep, tail, redirect) works perfectly with Docker output. That's the power of the Unix philosophy.
