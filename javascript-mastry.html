01

Variables — var, let, const

1️⃣

Concept Explanation

A variable is a named container that holds a value in memory. JavaScript has three keywords — var, let, const — each with different rules for scope, hoisting, and reassignment.

KeywordScopeHoistedRe-declarableReassignableTDZ
varFunction✅ as undefined✅ Yes✅ Yes❌ No
letBlock✅ uninitialized❌ No✅ Yes✅ Yes
constBlock✅ uninitialized❌ No❌ No✅ Yes

Internal Working — Two-Phase Execution

Phase 1 — Compilation (before any code runs):
┌──────────────────────────────────────────┐
│ var x → undefined │ ← hoisted & initialized
│ let y → <TDZ — not initialized> │ ← hoisted, NOT initialized
│ const z → <TDZ — not initialized> │ ← hoisted, NOT initialized
└──────────────────────────────────────────┘

Phase 2 — Execution (line by line):
values assigned when engine reaches those lines
⚠️ var leaks out of blocks

var does NOT respect {} blocks. It leaks into the nearest function scope. This is the #1 source of var bugs.

⚠️ const ≠ immutable

const prevents the binding from being reassigned. The object's properties can still change.

🧠 Mental Model — Labeled Boxes

var — A box in the whole room (function). Exists before you fill it. Works anywhere in the room.

let — A box on one shelf (block). Cannot leave the shelf. You can refill it.

const — A permanently labeled box. You can't re-label it, but you can rearrange what's inside (for objects/arrays).

2️⃣

Code Examples

JavaScriptBasic
// var — re-declarable, function scoped
var city = "Delhi";
var city = "Mumbai"; // ✅ no error

// let — block scoped, reassignable
let score = 0;
score = 100; // ✅

// const — binding cannot be reassigned
const PI = 3.14159;
// PI = 3; // ❌ TypeError
JavaScriptReal-World — Loop Bug
// ❌ var: all callbacks share same i
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log("var:", i), 100);
}
// Output: var: 3, var: 3, var: 3

// ✅ let: each iteration gets its own scoped i
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log("let:", i), 100);
}
// Output: let: 0, let: 1, let: 2

Common Mistakes

02

Primitive vs Reference Types

1️⃣

Concept Explanation

Every JavaScript value is either a primitive (stored by value) or a reference type (stored by reference). The difference determines how values are copied and compared.

CategoryTypesStored InCopied ByMutable?
Primitivestring number bigint boolean undefined null symbolStackValue❌ No
ReferenceObject Array Function Date Map SetHeap (pointer on stack)Reference✅ Yes

Internal Working — Stack vs Heap

STACK (fast, fixed-size) HEAP (large, dynamic)
┌──────────────────────┐ ┌───────────────────────────┐
│ a = 10 │ │ │
│ b = 10 ← full copy │ │ { name: "Vipin" } ←─────┼── user1
│ │ │ (same object) ←─────┼── user2
│ user1 = [ref:0x001] ─┼──────────►│ [address: 0x001] │
│ user2 = [ref:0x001] ─┼──────────►│ │
└──────────────────────┘ └───────────────────────────┘

Changing b does NOT affect a (different stack slots)
Changing user2.name DOES affect user1 (same heap object)
🧠 Mental Model

Primitive = photocopy. Give someone a copy — they mark it up, your original is fine.

Reference = Google Doc link. Share the link — they edit it, and you both see the changes.

2️⃣

Code Examples

JavaScriptBasic → Advanced
// PRIMITIVES — copied by value
let a = 10;
let b = a;        // b gets a full COPY of 10
b = 99;
console.log(a);   // 10 — unchanged
console.log(b);   // 99

// REFERENCE — copied by pointer
let user1 = { name: "Vipin" };
let user2 = user1;           // same object in memory
user2.name = "Singh";
console.log(user1.name);     // "Singh" — user1 affected!

// COMPARISON
console.log({} === {});      // false — different objects
console.log([] === []);      // false — different arrays
console.log("hi" === "hi"); // true  — same primitive value

// SHALLOW COPY — spread operator
const original = { score: 100 };
const clone = { ...original };
clone.score = 0;
console.log(original.score);  // 100 — untouched ✅

// DEEP COPY — JSON (flat objects only)
const deep = { user: { prefs: { theme: "dark" } } };
const copy = JSON.parse(JSON.stringify(deep));
copy.user.prefs.theme = "light";
console.log(deep.user.prefs.theme); // "dark" — untouched ✅
// ⚠️ JSON.stringify drops: functions, undefined, Dates, Symbols

// REAL-WORLD: Settings draft isolation
const liveSettings = { theme: "dark", fontSize: 16 };
let draft = { ...liveSettings };     // isolated copy
draft.theme = "light";              // editing draft doesn't affect live
// On save:
// Object.assign(liveSettings, draft);

📌 When does undefined appear automatically?

🧠 Mental Model

undefined = An empty mailbox. Nothing was ever put there.

null = An empty mailbox with a sign saying "Empty on purpose."

2️⃣

Code Examples

JavaScriptAll Cases

console.log(x);                          // undefined

console.log(obj.b);                      // undefined — no property

// null — set by developer
let currentUser = null;                  // intentional: no user logged in
currentUser = { name: "Vipin" };        // now assigned

// The equality trap
console.log(null == undefined);         // true  — loose eq
console.log(null === undefined);        // false — strict eq
console.log(typeof null);               // "object" ← JS bug!
console.log(typeof undefined);          // "undefined"

// Safe check for BOTH null AND undefined
function process(value) {
  if (value == null) {                   // catches null AND undefined
    console.log("No value provided");
    return;
  }
  console.log(value);
}
process(null);       // "No value provided"
process(undefined);  // "No value provided"
process(42);         // 42

// JSON behaviour difference
console.log(JSON.stringify({ a: undefined, b: null }));
// {"b":null}  — undefined is omitted, null is kept

The Global Object

1️⃣

Concept Explanation

The global object is the top-level container in a JavaScript runtime. Every built-in (Math, JSON, setTimeout) and every var declared at the top level lives on it.

EnvironmentGlobal ObjectUniversal (ES2020+)
BrowserwindowglobalThis
Node.jsglobal
Web Workerself
📌 Key Behavior
  • var declarations at top level → become properties of the global object
  • let / const at top level → do NOT become global object properties
  • All built-ins (Math, parseInt, setTimeout) live on the global object
JavaScriptBrowser
var appName = "MyApp";
console.log(window.appName);         // "MyApp" — on window!

let version = "1.0";
console.log(window.version);         // undefined — not on window

// globalThis works everywhere
console.log(globalThis.Math.PI);     // 3.14...
console.log(globalThis.parseInt("42")); // 42

// Environment detection
const isBrowser = typeof window !== "undefined";
const isNode = typeof process !== "undefined" && process.versions?.node;
console.log(isBrowser ? "Browser" : "Node");

function getAge(input) {

  return parsed;
}

Common Mistakes

❌ Using == instead of ===

Loose equality triggers coercion. userInput == 0 is true for "", false, null, "0". Always use ===.

❌ Assuming [] and {} are falsy

❌ parseInt without radix

parseInt("010") could be 8 (octal) in some environments. Always: parseInt("010", 10).

JavaScriptPrecedence in Action
// Arithmetic
2 + 3 * 4       // 14 — * before +
(2 + 3) * 4     // 20 — () overrides

// Right-to-left: exponentiation
2 ** 3 ** 2     // 512 — 2**(3**2) = 2**9, NOT (2**3)**2 = 64!

// && has higher precedence than ||
true || false && false  // true — evaluated as: true || (false && false)

"1" + 2 + 3    // "123" — "1"+"2" = "12", "12"+"3" = "123"
1 + 2 + "3"    // "33"  — 1+2 = 3, 3+"3" = "33"

// typeof binds tighter than +
typeof 1 + "2"  // "number2" — (typeof 1) + "2"
typeof (1 + "2") // "string"  — typeof("12")

// Assignment is right-to-left
let a, b, c;
a = b = c = 5;  // c=5, b=5, a=5 ✅

// Real-world: always use parens for complex logic
// ❌ Ambiguous

// ✅ Explicit intent

1 === 1              // true

null > 0             // false
null == 0            // false — null only == null/undefined

Nullish Coalescing ?? and Optional Chaining ?.

JavaScriptES2020 Operators
// ?? — returns right side ONLY if left is null or undefined
0 || "default"       // "default" — 0 is falsy, || triggers
0 ?? "default"       // 0         — 0 is not null/undefined, ?? keeps it
null ?? "fallback"  // "fallback"
"" ?? "fallback"    // ""         — empty string kept by ??

// user.name              // ❌ TypeError

config?.db?.host         // "localhost"
config?.cache?.host      // undefined — safe
config?.db?.connect?.()  // undefined — optional method call

P2·04

Short-Circuiting

1️⃣

Concept Explanation

Short-circuiting means JavaScript stops evaluating an expression as soon as the result is determined. The right side may never execute.

&& → stops at FIRST FALSY → returns that falsy value (or last if all truthy)
|| → stops at FIRST TRUTHY → returns that truthy value (or last if all falsy)
?? → stops at FIRST non-null/undefined → returns it

Return values are NOT necessarily boolean — they return the actual operand value
JavaScriptAll Patterns
// Side effect test — right side may never run
let count = 0;
false && count++;   // count stays 0 — right side NOT evaluated
true && count++;    // count becomes 1 — right side IS evaluated

// Pattern 1: Conditional execution
user.isAdmin && showAdminPanel(); // only runs if isAdmin is truthy

// Pattern 2: Default value with || (triggers on ANY falsy)
function greet(name) {
  name = name || "Guest";    // "" triggers this — careful!
  return `Hello, ${name}!`;
}

// Pattern 3: Default value with ?? (only null/undefined)
function greet2(name) {
  name = name ?? "Guest";    // "" is preserved ✅
  return `Hello, ${name}!`;
}

// Pattern 4: Lazy initialization / memoization
let cache = null;
function getData() {
  return cache || (cache = fetchFromServer()); // fetches once, then cached
}

// Pattern 5: Conditional rendering (React-style)
const isLoggedIn = true;
const ui = isLoggedIn && '<div>Dashboard</div>'; // shows div

// ❌ Danger: number as condition renders "0" in React
const count = 0;
const bad = count && '<span>items</span>'; // renders 0, not nothing!
// ✅ Fix: use boolean condition
const good = count > 0 && '<span>items</span>';
3️⃣

Common Mistakes

❌ Number condition in JSX/templates renders "0"

When count is 0, count && <span> renders the number 0 to the DOM, not nothing. Always use count > 0 && or a ternary.

❌ Side effects inside short-circuit — hard to debug

false && (x = 10) — the assignment NEVER runs. If you depend on a side effect, don't hide it inside a short-circuit expression.

P3·01

if / switch

1️⃣

Concept Explanation

if evaluates any truthy/falsy expression and conditionally runs a block. switch evaluates one expression once and matches it against case values using strict equality (===) — no type coercion.

if / else if / elseswitch
ComparisonAny expression / truthy-falsyStrict equality === only
Range checks✅ Yes❌ No (use switch(true) pattern)
Fall-through❌ No✅ Yes (requires break to stop)
Best forComplex conditions, rangesSingle value vs many exact values
⚠️ switch uses === internally

switch("1") with case 1: will not match — "1" !== 1. No coercion happens.

📌 Internal: How switch works

JS evaluates the switch expression once, then walks the case list doing === comparison. When found, it executes downward until it hits break or the end. This downward execution is called fall-through.

🧠 Mental Model

if = A series of gates — check each gate until one opens.

switch = A lookup table — jump to the matching row, execute downward until you hit a stop sign (break).

P3·02

Loops

1️⃣

Concept Explanation

LoopUse WhenWorks On
forKnown number of iterationsCounters, indexes
whileUnknown iterations, condition drivenAny condition

📌 for...of — The Iterator Protocol

for...of calls [Symbol.iterator]() on the value. Any object implementing this protocol is iterable. Plain {} objects do NOT implement it — hence for...of throws on them. Use Object.entries() to loop object key-value pairs.

⚠️ for...in walks the prototype chain

If you add a property to Array.prototype, for...in on an array will include it. Never use for...in on arrays. Use for...of or forEach instead.

🧠 Mental Model

for = You know exactly how many laps to run.

while = Run until the traffic light turns red.

do...while = Step through the door first, THEN check if you should keep going.

for...of = Pick each item from a conveyor belt (values).

for...in = Read every label on every drawer — including inherited ones. Be careful!

P3·03

Control Flow — break, continue, throw, Labels

1️⃣

Concept Explanation

KeywordWhat it doesUsed in
breakExit the enclosing loop or switch immediatelyfor, while, switch
continueSkip rest of current iteration, go to nextfor, while, do...while
throwThrow an exception — unwinds the call stackAnywhere
Label + breakExit a specific named outer loopNested loops
Label + continueSkip to next iteration of a named outer loopNested loops
📌 How throw works internally

When throw executes, JS unwinds the call stack, looking for the nearest try...catch. If found, the catch block runs. The finally block always runs — even if there's a return inside try or catch.

try → throws error → catch → finally
try → return "value" → finally STILL runs → THEN returns "value"
try → return → finally → return "new" → overrides original return! ⚠️
2️⃣

Code Examples

JavaScriptbreak & continue

  if (i === 5) break;
  console.log(i); // 0 1 2 3 4
}

for (let i = 0; i < 10; i++) {

  console.log(divide(10, 2));  // 5

class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = "ValidationError";
    this.field = field;
  }
}

  constructor(message, statusCode) {
    super(message);
    this.name = "NetworkError";
    this.statusCode = statusCode;
  }
}

  if (!user.name)  throw new ValidationError("Name required", "name");
  if (!user.email) throw new ValidationError("Email required", "email");
}

try {
  validateUser({ name: "", email: "" });
} catch (err) {
  if (err instanceof ValidationError) {
    console.log(`Field '${err.field}': ${err.message}`);
  } else {
    throw err; // re-throw errors you can't handle
  }
}

// finally always runs — even with return
function withCleanup() {
  try {
    console.log("trying");
    return "from try";    // return queued
  } finally {
    console.log("finally"); // runs BEFORE return delivers
  }
}
console.log(withCleanup());
// "trying" → "finally" → "from try"

// Real-world: Retry with exponential backoff
async function fetchWithRetry(url, retries = 3) {
  for (let attempt = 1; attempt <= retries; attempt++) {
    try {
      const res = await fetch(url);
      if (!res.ok) throw new NetworkError("Bad response", res.status);
      return await res.json();
    } catch (err) {
      if (err instanceof NetworkError && err.statusCode === 404) {
        throw err; // never retry 404s
      }
      if (attempt === retries) throw err; // retries exhausted
      const delay = 2 ** attempt * 100; // 200ms, 400ms, 800ms
      console.log(`Retry ${attempt} in ${delay}ms...`);
      await new Promise(r => setTimeout(r, delay));
    }
  }
}

// deepFreeze utility
function deepFreeze(obj) {
  Object.keys(obj).forEach(key => {
    if (typeof obj[key] === "object" && obj[key] !== null) {
      deepFreeze(obj[key]);
    }
  });
  return Object.freeze(obj);
}

P4·03

The this Keyword

1️⃣

Concept Explanation

this refers to the object currently executing the function. For regular functions, it is NOT fixed at definition time — it is determined at call time based on how the function is called. This is the most misunderstood concept in JavaScript.

PriorityRuleHow it's calledthis =
1 (highest)New bindingnew Fn()New object created by new
2Explicit bindingfn.call(obj) / fn.apply(obj) / fn.bind(obj)()obj
3Implicit bindingobj.method()obj (before the dot)
4 (lowest)Default bindingfn()window or undefined (strict)
📌 Arrow Functions — Lexical this

Arrow functions do not have their own this. They inherit this from their lexical (surrounding) scope at definition time. This makes them perfect for callbacks inside methods, but wrong for object methods themselves.

🧠 Mental Model

Regular function: this = whoever is calling the function right now (at runtime).

Arrow function: this = whoever owned the context where the function was written (at definition time).

2️⃣

Code Examples

JavaScriptAll 4 Binding Rules
// RULE 4 — Default binding (lowest priority)
function showThis() { console.log(this); }
showThis(); // window (browser) or undefined (strict mode)

// RULE 3 — Implicit binding
const user = {
  name: "Vipin",
  greet() { console.log(this.name); }
};
user.greet(); // "Vipin" — this = user (the object before the dot)

// ⚠️ Losing implicit binding — THE classic trap
const fn = user.greet;  // detached from user
fn();                    // undefined — "user." context is GONE

// RULE 2 — Explicit binding
function greet() { return `Hello, ${this.name}`; }
const vipin = { name: "Vipin" };
const rahul = { name: "Rahul" };

greet.call(vipin);               // "Hello, Vipin"  (args: thisArg, a, b...)
greet.apply(rahul);              // "Hello, Rahul"  (args: thisArg, [array])
const boundGreet = greet.bind(vipin); // returns NEW permanently bound fn
boundGreet();                    // "Hello, Vipin" — binding cannot be changed
boundGreet.call(rahul);          // still "Vipin" — bind wins over call!

// RULE 1 — New binding (highest priority)
function User(name) {
  this.name = name;   // this = newly created object
}
const u = new User("Vipin"); // new: creates obj, sets this, returns obj
console.log(u.name);  // "Vipin"

// ARROW FUNCTION — lexical this
const timer = {
  name: "Vipin",
  start() {
    // Regular function — loses this in callback
    setTimeout(function() {
      console.log(this.name); // undefined — this = window
    }, 100);
    // Arrow function — inherits this from start()
    setTimeout(() => {
      console.log(this.name); // "Vipin" ✅
    }, 100);
  }
};
timer.start();

// Real-world: fixing this in class event handlers
class Counter {
  constructor() {
    this.count = 0;
    this.increment = this.increment.bind(this); // ✅ Fix 1: bind in constructor
  }
  increment() {
    this.count++;
    console.log(this.count);
  }
}
// button.addEventListener("click", counter.increment); // ✅ works now
