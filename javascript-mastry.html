01

Variables — var, let, const

1️⃣

Concept Explanation

A variable is a named container that holds a value in memory. JavaScript has three keywords — var, let, const — each with different rules for scope, hoisting, and reassignment.

KeywordScopeHoistedRe-declarableReassignableTDZ
varFunction✅ as undefined✅ Yes✅ Yes❌ No
letBlock✅ uninitialized❌ No✅ Yes✅ Yes
constBlock✅ uninitialized❌ No❌ No✅ Yes

Internal Working — Two-Phase Execution

Phase 1 — Compilation (before any code runs):
┌──────────────────────────────────────────┐
│ var x → undefined │ ← hoisted & initialized
│ let y → <TDZ — not initialized> │ ← hoisted, NOT initialized
│ const z → <TDZ — not initialized> │ ← hoisted, NOT initialized
└──────────────────────────────────────────┘

Phase 2 — Execution (line by line):
values assigned when engine reaches those lines
⚠️ var leaks out of blocks

var does NOT respect {} blocks. It leaks into the nearest function scope. This is the #1 source of var bugs.

⚠️ const ≠ immutable

const prevents the binding from being reassigned. The object's properties can still change.

🧠 Mental Model — Labeled Boxes

var — A box in the whole room (function). Exists before you fill it. Works anywhere in the room.

let — A box on one shelf (block). Cannot leave the shelf. You can refill it.

const — A permanently labeled box. You can't re-label it, but you can rearrange what's inside (for objects/arrays).

2️⃣

Code Examples

JavaScriptBasic
// var — re-declarable, function scoped
var city = "Delhi";
var city = "Mumbai"; // ✅ no error

// let — block scoped, reassignable
let score = 0;
score = 100; // ✅

// const — binding cannot be reassigned
const PI = 3.14159;
// PI = 3; // ❌ TypeError
JavaScriptReal-World — Loop Bug

  setTimeout(() => console.log("var:", i), 100);
}
// Output: var: 3, var: 3, var: 3

// ✅ let: each iteration gets its own scoped i

  setTimeout(() => console.log("let:", i), 100);
}
// Output: let: 0, let: 1, let: 2

Primitive vs Reference Types

1️⃣

Concept Explanation

Every JavaScript value is either a primitive (stored by value) or a reference type (stored by reference). The difference determines how values are copied and compared.

CategoryTypesStored InCopied ByMutable?
Primitivestring number bigint boolean undefined null symbolStackValue❌ No
ReferenceObject Array Function Date Map SetHeap (pointer on stack)Reference✅ Yes

Internal Working — Stack vs Heap

STACK (fast, fixed-size) HEAP (large, dynamic)
┌──────────────────────┐ ┌───────────────────────────┐
│ a = 10 │ │ │
│ b = 10 ← full copy │ │ { name: "Vipin" } ←─────┼── user1
│ │ │ (same object) ←─────┼── user2
│ user1 = [ref:0x001] ─┼──────────►│ [address: 0x001] │
│ user2 = [ref:0x001] ─┼──────────►│ │
└──────────────────────┘ └───────────────────────────┘

Changing b does NOT affect a (different stack slots)
Changing user2.name DOES affect user1 (same heap object)
🧠 Mental Model

Primitive = photocopy. Give someone a copy — they mark it up, your original is fine.

Reference = Google Doc link. Share the link — they edit it, and you both see the changes.

2️⃣

Code Examples

JavaScriptBasic → Advanced
// PRIMITIVES — copied by value
let a = 10;
let b = a;        // b gets a full COPY of 10
b = 99;
console.log(a);   // 10 — unchanged
console.log(b);   // 99

// REFERENCE — copied by pointer
let user1 = { name: "Vipin" };
let user2 = user1;           // same object in memory
user2.name = "Singh";
console.log(user1.name);     // "Singh" — user1 affected!

// COMPARISON
console.log({} === {});      // false — different objects
console.log([] === []);      // false — different arrays
console.log("hi" === "hi"); // true  — same primitive value

// SHALLOW COPY — spread operator
const original = { score: 100 };
const clone = { ...original };
clone.score = 0;
console.log(original.score);  // 100 — untouched ✅

// DEEP COPY — JSON (flat objects only)
const deep = { user: { prefs: { theme: "dark" } } };
const copy = JSON.parse(JSON.stringify(deep));
copy.user.prefs.theme = "light";
console.log(deep.user.prefs.theme); // "dark" — untouched ✅
// ⚠️ JSON.stringify drops: functions, undefined, Dates, Symbols

// REAL-WORLD: Settings draft isolation
const liveSettings = { theme: "dark", fontSize: 16 };
let draft = { ...liveSettings };     // isolated copy
draft.theme = "light";              // editing draft doesn't affect live
// On save:
// Object.assign(liveSettings, draft);

Concept Explanation

EnvironmentGlobal ObjectUniversal (ES2020+)

📌 Key Behavior
  • var declarations at top level → become properties of the global object

// Arithmetic

// Assignment is right-to-left
let a, b, c;

Short-Circuiting

1️⃣

Concept Explanation

Short-circuiting means JavaScript stops evaluating an expression as soon as the result is determined. The right side may never execute.

&& → stops at FIRST FALSY → returns that falsy value (or last if all truthy)
|| → stops at FIRST TRUTHY → returns that truthy value (or last if all falsy)
?? → stops at FIRST non-null/undefined → returns it

Return values are NOT necessarily boolean — they return the actual operand value
JavaScriptAll Patterns

// Pattern 2: Default value with || (triggers on ANY falsy)

  name = name || "Guest";    // "" triggers this — careful!
  return `Hello, ${name}!`;
}

// Pattern 3: Default value with ?? (only null/undefined)
function greet2(name) {
  name = name ?? "Guest";    // "" is preserved ✅
  return `Hello, ${name}!`;
}

// Pattern 4: Lazy initialization / memoization

  return cache || (cache = fetchFromServer()); // fetches once, then cached
}

// ❌ Danger: number as condition renders "0" in React

P3·01

1️⃣

Concept Explanation

if evaluates any truthy/falsy expression and conditionally runs a block. switch evaluates one expression once and matches it against case values using strict equality (===) — no type coercion.

if / else if / elseswitch
ComparisonAny expression / truthy-falsyStrict equality === only
Range checks✅ Yes❌ No (use switch(true) pattern)
Fall-through❌ No✅ Yes (requires break to stop)
Best forComplex conditions, rangesSingle value vs many exact values
⚠️ switch uses === internally

switch("1") with case 1: will not match — "1" !== 1. No coercion happens.

📌 Internal: How switch works

JS evaluates the switch expression once, then walks the case list doing === comparison. When found, it executes downward until it hits break or the end. This downward execution is called fall-through.

🧠 Mental Model

if = A series of gates — check each gate until one opens.

switch = A lookup table — jump to the matching row, execute downward until you hit a stop sign (break).

📌 for...of — The Iterator Protocol

for...of calls [Symbol.iterator]() on the value. Any object implementing this protocol is iterable. Plain {} objects do NOT implement it — hence for...of throws on them. Use Object.entries() to loop object key-value pairs.

📌 How throw works internally

When throw executes, JS unwinds the call stack, looking for the nearest try...catch. If found, the catch block runs. The finally block always runs — even if there's a return inside try or catch.

try → throws error → catch → finally
try → return "value" → finally STILL runs → THEN returns "value"

// Real-world: Retry with exponential backoff
async function fetchWithRetry(url, retries = 3) {
  for (let attempt = 1; attempt <= retries; attempt++) {
    try {
      const res = await fetch(url);
      if (!res.ok) throw new NetworkError("Bad response", res.status);
      return await res.json();
    } catch (err) {
      if (err instanceof NetworkError && err.statusCode === 404) {
        throw err; // never retry 404s
      }
      if (attempt === retries) throw err; // retries exhausted
      const delay = 2 ** attempt * 100; // 200ms, 400ms, 800ms
      console.log(`Retry ${attempt} in ${delay}ms...`);
      await new Promise(r => setTimeout(r, delay));
    }
  }
}

1️⃣

Concept Explanation

this refers to the object currently executing the function. For regular functions, it is NOT fixed at definition time — it is determined at call time based on how the function is called. This is the most misunderstood concept in JavaScript.

Regular function: this = whoever is calling the function right now (at runtime).

P5·01

Array Internals

1️⃣

Concept Explanation

Simple Explanation

An array is an ordered list of values. You access items by their position (index), starting at 0. Arrays can hold any mix of types.

MDN-Level Explanation

In JavaScript, arrays are objects whose keys happen to be non-negative integers (indices). They inherit from Array.prototype and carry a special length property that automatically tracks the highest index + 1. Because they are objects, you can also attach arbitrary named properties to them — though doing so is almost always a mistake.

Internal Working — V8 Element Kinds

V8 (Chrome/Node JS engine) optimises arrays using element kinds. It picks a compact internal representation based on what the array contains and downgrades (never upgrades) as elements change:

KindWhat triggers itMemory layout
FAST_SMI_ELEMENTSAll integersContiguous tagged integers — fastest
FAST_DOUBLE_ELEMENTSMix of integers & floatsUnboxed 64-bit doubles
FAST_ELEMENTSAny object / mixed typesPointer array to heap objects
DICTIONARY_ELEMENTSSparse (holes) / huge indicesHash map — much slower
Dense array — contiguous block in memory (fast):
[0] = 10 │ [1] = 20 │ [2] = 30

Sparse array — dictionary, gaps exist (slow):
arr[0] = 10 │ arr[1] = <hole> │ arr[999999] = 1

Mental Model

Think of an array as a numbered shelf
Slot 0, slot 1, slot 2 … Each slot can hold anything. The engine tries to keep all slots packed tightly in RAM for speed. The moment you skip slots or mix types wildly, it falls back to a slower hash-map mode. Keep arrays dense and type-consistent for maximum performance.

P5·03

Mutability Behavior

1️⃣

Concept Explanation

Simple Explanation

Some array methods change the original array (mutating). Others return a new array and leave the original untouched (non-mutating). Knowing which is which prevents subtle bugs, especially when arrays are shared across functions.

MDN-Level Explanation

Arrays are reference types. When you pass an array to a function or assign it to a new variable, you copy the reference, not the data. Mutating methods (like push, splice, sort) modify the backing store in-place. Non-mutating methods (like map, filter, slice) allocate a new array and leave the original untouched.

Mutating (changes original)Non-mutating (returns new)
push, pop, shift, unshiftmap, filter, reduce
splice, fill, copyWithinslice, concat, flat, flatMap
sort, reversetoSorted, toReversed (ES2023)

Mental Model

Reference = shared pointer
Think of an array stored in a box. A variable holds a label pointing to that box. If you hand someone your label (const b = a), they can open the same box and rearrange things inside. A non-mutating method makes a photocopy of the box's contents and hands you that instead — the original box is untouched.

1️⃣

Concept Explanation

Simple Explanation

Scope & Lexical Environment

1️⃣

Concept Explanation

Simple Explanation

Scope is the set of variables a piece of code can "see". When you look up a variable, JavaScript starts in the current scope and walks outward until it finds the binding or hits the global scope. This walk is called the scope chain.

Every time a function is called, the engine creates an Execution Context which holds a Lexical Environment — a data structure with two parts: an Environment Record (all local bindings) and a pointer to the outer Lexical Environment. Variable resolution follows this chain at definition time (lexical scoping), not call time. This means where a function is written determines what it can see, not where it is called from.

1️⃣

Concept Explanation

Simple Explanation

A closure is when a function "remembers" the variables from its surrounding scope even after that outer scope has finished executing. The inner function carries those variables with it wherever it goes.

A closure is the combination of a function and the Lexical Environment in which it was defined. When an inner function references a variable from an outer function, the JS engine keeps that outer Lexical Environment alive in memory — even after the outer function returns — because the inner function holds a reference to it. The outer function's stack frame is gone, but its bindings live on the heap, reachable only through the closure.

└─────────────────────────────────────────┘

A function with a backpack
When a function is born inside another function, it gets a backpack containing all the variables it references from its birth environment. It carries this backpack everywhere.     if (cache.has(key)) return cache.get(key); // cache hit
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

const expensiveCalc = memoize((n) => {
  console.log("computing...");
  return n * n;
});

expensiveCalc(5); // logs "computing..." → 25
expensiveCalc(5); // cache hit → 25 (no log)
expensiveCalc(6); // logs "computing..." → 36

1️⃣

Concept Explanation

Simple Explanation

Arrow functions are a shorter syntax for writing functions introduced in ES6. Beyond the compact syntax, they have a key behavioural difference: they do not have their own this — they inherit this from the surrounding lexical scope where they were defined.

MDN-Level Explanation

Arrow functions are anonymous function expressions that differ from regular functions in four ways:

  • No own this — this is resolved lexically from the enclosing scope at definition time
  • No arguments object — use rest parameters instead
  • Cannot be used as constructors — new (() => {}) throws TypeError
  • No prototype property

Because this is fixed at definition time, arrow functions are ideal for callbacks inside methods where you need to preserve the method's this.

P7·01

Prototype Chain

1️⃣

Concept Explanation

Simple Explanation

Every JavaScript object has a hidden link to another object called its prototype. When you access a property that doesn't exist on the object itself, JS follows that link and looks on the prototype — then the prototype's prototype — all the way up to null. This chain of links is the prototype chain.

MDN-Level Explanation

Each object has an internal slot [[Prototype]] (accessible via Object.getPrototypeOf(obj) or the legacy __proto__). Property lookup is a runtime walk up this chain: the engine checks the object's own properties first, then delegates to [[Prototype]], then its [[Prototype]], terminating when null is reached. If the property is never found, the result is undefined.

Internal Working — Property Lookup Algorithm

obj.greet() ← JS engine lookup order:

1. Check obj's own properties → not found
2. Follow [[Prototype]] → check that object → not found
3. Follow [[Prototype]] again → found! → call it
4. If we reach null → return undefined

obj → Animal.prototype → Object.prototype → null

Setting a property always writes directly on obj — never on its prototype. Only reads walk the chain. This is called shadowing.

Mental Model

🧠 Think of it as a chain of reference books
You open your own notebook first (own properties). If the answer isn't there, you check the department library (prototype). If still not found, you check the central archive (Object.prototype). The chain ends when you find the answer — or run out of books (null). Writing always goes in your own notebook, never in the shared library.

P7·02

Inheritance Model

1️⃣

Concept Explanation

Simple Explanation

Prototypal inheritance means one object can delegate behavior to another. Instead of copying methods from a parent, a child object holds a reference ([[Prototype]]) to the parent. When you call a method the child doesn't own, JS walks up to the parent automatically.

MDN-Level Explanation

JavaScript implements differential inheritance — objects store only what makes them different from their prototype. When you call new Constructor(), the engine creates a fresh object whose [[Prototype]] is set to Constructor.prototype, then calls the constructor function with this pointing to that new object. This wires the chain: instance → Constructor.prototype → Object.prototype → null.

Internal Working — What new Does

new Animal("Rex") internally does:

1. Create → obj = Object.create(Animal.prototype)
2. Bind → call Animal.call(obj, "Rex") (this = obj)
3. Return → if constructor returns non-object, return obj

Result chain: obj → Animal.prototype → Object.prototype → null

Mental Model

🧠 Delegation, not copying
Classical OOP (Java/C++) copies methods into each instance at class-load time. JavaScript instances carry no methods — only a pointer to a shared prototype. It's like every employee having a link to the company handbook, not a personal photocopy of it. The handbook can be updated and everyone benefits immediately.

P7·03

JS vs Classical OOP

1️⃣

Concept Explanation

Simple Explanation

Classical OOP (Java, C++) uses classes as blueprints — objects are stamped out from a template and carry copied methods. JavaScript uses prototype chains — objects link to other objects and delegate missing behavior at runtime. There are no true classes in the engine; ES6 class syntax is a convenient layer on top of the same prototype mechanism.

MDN-Level Explanation

In classical OOP, inheritance is a compile-time copy operation: the child class's vtable includes copies of or pointers to parent methods, determined before the program runs. JavaScript's inheritance is purely dynamic: property lookup traverses the live prototype chain at access time. This means you can modify a prototype after creating instances and all existing instances immediately reflect the change — something impossible in compiled classical OOP.

Method lookupStatic — resolved at compile timeDynamic — walked at runtime

Multiple inheritanceVia interfaces / mixinsVia mixin pattern (Object.assign to prototype)
Private stateLanguage keyword (private)Closures, WeakMap, or # fields (ES2022)

Classical OOP: a class is a photocopied blueprint. Each new object gets its own full copy. Changes to the blueprint after printing don't affect existing copies.

JavaScript: a prototype is a shared Google Doc. All objects link to it. Edit the doc — every linked object sees the new content immediately. The power and the danger are the same thing.

P8·01

Class Syntax vs Prototypes

1️⃣

Concept Explanation

Simple Explanation

The class keyword introduced in ES6 is syntactic sugar over JavaScript's existing prototype-based inheritance. It looks like Java or Python classes, but under the hood the engine still uses constructor functions and .prototype chains. Nothing about the object model changed — only the notation.

P8·02

Constructor Behavior

1️⃣

Concept Explanation

Simple Explanation

The constructor method inside a class is the function that runs when you call new ClassName(). It receives the arguments you pass, sets up the object's own properties via this, and implicitly returns the new instance.

MDN-Level Explanation

Every class has exactly one constructor. If omitted, an empty one is synthesised. For a derived class (one that uses extends), the constructor must call super() before any access to this — otherwise a ReferenceError is thrown. super() invokes the parent's constructor, which allocates the object and sets up the parent chain before the child can add its own properties. A constructor may return a plain object explicitly, which overrides the default this-return; returning a primitive is ignored.

Internal Working — new Step-by-Step

new Dog("Rex", "Labrador") — what the engine does: Step 1: Create a brand-new empty object obj = {} Step 2: Set its prototype → obj.[[Prototype]] = Dog.prototype Step 3: Bind this to obj inside the constructor Step 4: Execute constructor body (super() first for subclasses) Dog calls super("Rex") → runs Animal.constructor on obj then sets obj.breed = "Labrador" Step 5: Return obj (unless constructor explicitly returns another object)

Inheritance with super()

In a derived class, this is not available until super() completes. The parent constructor is responsible for creating the object. Accessing this before super() throws a ReferenceError.

Private Fields

1️⃣

Concept Explanation

Simple Explanation

Private fields use a # prefix (e.g., #balance) and are truly inaccessible from outside the class body. They cannot be read, written, or even checked for existence from external code — not even by subclasses.

MDN-Level Explanation

Private class fields are part of the class body declaration and are stored in an internal [[PrivateFields]] slot of each instance. Unlike the old convention of prefixing with _ (which is just a naming hint, not enforced), # fields are enforced by the engine: accessing instance.#field outside the class body throws a SyntaxError at parse time, not a runtime error. Private fields must be declared at the top of the class body before use.

Internal Working

Public field: stored as own property on the instance → visible via Object.keys, JSON.stringify Private field: stored in [[PrivateFields]] slot → invisible to any outside code instance.#balance → SyntaxError (parse-time, not runtime!) instance["#balance"] → undefined (not the same thing at all!) The engine uses the class itself as the "brand" — only methods defined in that exact class body can access its private fields.

P9·01

ES Modules

1️⃣

Concept Explanation

Simple Explanation

Before ES6, JavaScript had no built-in way to split code across files — developers relied on <script> tags in a specific order or bundlers like Browserify. ES Modules (ESM) give JavaScript a native, standardised module system: each file is its own sealed scope and can explicitly declare what it exposes (export) and what it needs (import).

MDN-Level Explanation

An ES Module is any JavaScript file treated by the runtime as a module record. In browsers a <script type="module"> tag triggers this; in Node.js either a .mjs extension or "type":"module" in package.json does the same. Module records are parsed, linked, and evaluated by the engine in three distinct phases. Key guarantees the spec provides:

  • Module code is always in strict mode — no opt-in required.
  • Each module has its own module scope; top-level declarations are not globals.
  • A module is evaluated exactly once, no matter how many files import it (singleton semantics).
  • Imports are live read-only bindings to the exporting module's namespace, not copies.
  • The dependency graph is resolved statically before any code executes (enabling tree-shaking by bundlers).

Internal Working — Three-Phase Module Loading

Phase 1 — Construction (parse) Find every static import/export statement. Download and parse each dependency file recursively. Build a Module Record for each file — no code runs yet. Phase 2 — Linking (connect live bindings) Create the module namespace object for each record. Wire exported names to memory cells (slots). Imported names become read-only views of those same slots. Phase 3 — Evaluation (run) Execute module bodies in post-order (dependencies before dependants). Fill the memory cells with real values. Because a module runs only once, the namespace is shared across all importers.

Import / Export Syntax

1️⃣

Concept Explanation

Simple Explanation

Every binding a module wants to share must be explicitly exported; every binding another module wants to use must be explicitly imported. Nothing leaks automatically. There are two flavours: named (many exports per file, imported by exact name) and default (one primary export per file, imported with any name the consumer chooses).

MDN-Level Explanation

The export keyword marks a binding as part of the module's public namespace. The import declaration creates a live read-only binding in the importing module that points to the original memory cell. Because the link is live, if the exporting module later reassigns its variable, the importing module sees the updated value without re-importing. Default exports are just named exports under the reserved name "default"; import Foo from '…' is sugar for import { default as Foo } from '…'.

Module Scope & Loading

1️⃣

Concept Explanation

Simple Explanation

Every ES module runs in its own private scope. Nothing at the top level leaks to the global object. Imports are resolved before any code runs — you cannot use an import inside an if block or a function (except with import(), the dynamic form). When you need to load a module conditionally or lazily at runtime, you use the dynamic import expression import('...') which returns a Promise.

MDN-Level Explanation

Static import declarations are parsed at compile time; the engine builds a full module graph before evaluation begins. This is what enables tree-shaking — a bundler can determine statically which exports are used and drop the rest. Dynamic import() is an expression (not a declaration) that initiates a new asynchronous module load at runtime, returning a Promise that resolves to the module namespace object. It can appear anywhere an expression is valid, enabling code-splitting and lazy loading.

Internal Working — Static vs Dynamic Loading

Static import (declaration) Resolved at parse time — before any code executes. Specifier must be a string literal (no variables). Enables tree-shaking and compile-time optimisation. Always runs at the top of the module, regardless of position in file. import { foo } from './foo.js'; // ← resolved statically Dynamic import() (expression) Resolved at runtime, returns Promise<ModuleNamespace>. Specifier can be any expression (variable, template literal). Module is fetched, parsed, and evaluated lazily on first call. Subsequent calls return the cached module record (no re-evaluation). const { foo } = await import('./foo.js'); // ← resolved at runtime

Mental Model

Static import = blueprint wiring; dynamic import() = on-demand delivery
Static imports are like the electrical wiring in a building — everything is connected before anyone moves in. 1️⃣

1️⃣

Concept Explanation

JSON (JavaScript Object Notation) is a text-based data interchange format. JavaScript's built-in JSON object provides two methods: JSON.stringify() to convert a value to a JSON string, and JSON.parse() to convert a JSON string back to a JavaScript value.

Simple Explanation

MDN-Level Explanation

JSON.stringify(value, replacer, space) performs a depth-first traversal of the value. Non-serializable values (undefined, functions, Symbols, BigInt) are either omitted (in objects), converted to null (in arrays), or throw. If an object has a toJSON() method, it is called and its return value is serialized instead. JSON.parse(text, reviver) parses strictly valid JSON — single-quoted strings or trailing commas throw SyntaxError.

Mental Model

JSON is a subset of JavaScript literal syntax with strict rules: keys must be double-quoted, no functions, no comments, no trailing commas.

Internal Working

JS engines compile regex patterns into a finite automaton (NFA or DFA). Matching is linear for simple patterns. Certain patterns (nested quantifiers on overlapping groups) can cause catastrophic backtracking — O(2ⁿ) worst case — a real security issue known as ReDoS.

Typed Arrays & Binary Data

1️⃣

Concept Explanation

Typed arrays provide array-like views over raw binary memory. Unlike regular JS arrays (which are hash tables holding boxed values), typed arrays map directly onto a contiguous block of bytes — making them indispensable for WebGL, Web Audio, WebSockets, and file processing.

Simple Explanation

Imagine a plain JS array as a flexible bag of mixed items. A typed array is a strict row of numbered storage slots — each slot is exactly the same size (e.g., 4 bytes for Int32Array), packed tightly in memory.

Three-Layer Architecture

  • ArrayBuffer — the raw bytes. Just memory; you can't read or write it directly.
  • TypedArray — a typed view over an ArrayBuffer (Uint8Array, Float32Array, etc.).
  • DataView — a flexible view that lets you read/write different types at arbitrary byte offsets, useful for binary protocols.

P11·01

Iteration Protocols

1️⃣

Concept Explanation

JavaScript defines two complementary protocols — the iterable protocol and the iterator protocol — that together let any object plug into the language's built-in iteration machinery (for...of, spread, destructuring, Array.from, etc.).

Simple Explanation

Think of a playlist: the playlist itself is the iterable (it knows how to produce a player), and the player is the iterator (it keeps track of which song is playing and gives you the next one on demand). You press next → it gives you a song and remembers where it is.

MDN-Level Explanation

Iterable protocol: an object is iterable if it has a method at key Symbol.iterator that returns an iterator object.

Iterator protocol: an object is an iterator if it has a next() method that returns a plain object with two fields: { value: any, done: boolean }. When done is true the sequence is exhausted; any further calls must continue returning { value: undefined, done: true }.

An object can satisfy both protocols simultaneously (an iterator-iterable) by returning this from [Symbol.iterator](). Built-in iterators (array iterator, string iterator, Map iterator, etc.) all do this.

Internal Working

When the engine encounters for (const x of obj) it:

  1. Calls obj[Symbol.iterator]() to obtain an iterator.
  2. Repeatedly calls iterator.next().
  3. Assigns result.value to the loop variable.
  4. Stops when result.done === true.
  5. If the loop body exits early (break, return, uncaught throw), the engine calls iterator.return() (if defined) to allow cleanup.

Spread and destructuring follow the same steps internally. Array.from(obj) also calls [Symbol.iterator]() first and falls back to the array-like (.length + indexed access) path only if that method is absent.

Mental Model

Two interfaces, one contract:
Iterable = "I can produce a cursor."
Iterator = "I am a cursor; ask me for the next value."

Basic — manually calling the iterator protocol

const arr = [10, 20, 30];

Intermediate — what for…of does under the hood

function manualForOf(iterable) {
  const iterator = iterable[Symbol.iterator]();
  let result;
  while (!(result = iterator.next()).done) {
    console.log(result.value);
  }
}

manualForOf(['a', 'b', 'c']); // logs a  b  c
manualForOf('hi');          // logs h  i  (strings are iterable!)
manualForOf(new Set([1,2])); // logs 1  2

Real-World — spread & destructuring powered by the protocol

// spread uses [Symbol.iterator] internally

// destructuring uses [Symbol.iterator] internally

// Array.from uses it too
Array.from(map.keys()); // ['a', 'b']

P11·02

Implementing Iterable Objects

1️⃣

Concept Explanation

Any object can become iterable by adding a [Symbol.iterator]() method. This lets your custom data structures participate in for...of, spread, destructuring, and all other iteration-aware language features without any special engine support.

Simple Explanation

You're building a custom Range object that represents numbers from start to end. By adding [Symbol.iterator] you can write for (const n of new Range(1, 5)) instead of a manual loop.

MDN-Level Explanation

To implement the iterable protocol on a class or plain object, define a method with the computed key [Symbol.iterator]. The method must return an object that satisfies the iterator protocol (has a next() returning { value, done }). The simplest approach is to return an object with a closure over the mutable cursor state, or to return a generator (covered in P11·04).

Making the iterator also return this from its own [Symbol.iterator] makes it an iterator-iterable — useful so the iterator itself can be passed to for...of without wrapping.

Internal Working

The Symbol.iterator well-known symbol is a unique primitive key stored on the global Symbol object. When the engine needs an iterator it performs a spec operation called GetIterator: it looks up obj[Symbol.iterator], throws TypeError if it's not a function, then calls it with obj as this. The returned object is checked for a next method. State (cursor position) lives in the closure — the engine never sees or stores it directly.

Mental Model

[Symbol.iterator]() is a factory: it manufactures a fresh iterator (cursor) each time it's called. That's why you can have two independent for...of loops over the same iterable running simultaneously without them interfering.

2️⃣

Code Examples

Basic — Range iterable (closure-based iterator)

class Range {
  constructor(start, end) {
    this.start = start;
    this.end   = end;
  }

  [Symbol.iterator]() {
    let current = this.start;
    const end   = this.end;
    return {
      next() {
        return current <= end
          ? { value: current++, done: false }
          : { value: undefined, done: true };
      }
    };
  }
}

const r = new Range(1, 5);
for (const n of r) console.log(n); // 1 2 3 4 5
[...r];                              // [1, 2, 3, 4, 5]
const [a, b] = r;                    // a=1, b=2

Intermediate — linked-list iterable

class LinkedList {
  constructor() { this.head = null; }

  push(val) {
    this.head = { val, next: this.head };
  }

  [Symbol.iterator]() {
    let node = this.head;
    return {
      next() {
        if (!node) return { done: true, value: undefined };
        const value = node.val;
        node = node.next;
        return { value, done: false };
      }
    };
  }
}

const list = new LinkedList();
list.push(3); list.push(2); list.push(1);
[...list]; // [1, 2, 3] (head-first traversal)

Real-World — infinite counter with early-exit via return()

class Counter {
  constructor(from = 0) { this.from = from; }

  [Symbol.iterator]() {
    let n = this.from;
    return {
      next()   { return { value: n++, done: false }; },
      return() { console.log('cleanup!'); return { done: true }; }
    };
  }
}

for (const n of new Counter()) {
  if (n === 3) break; // triggers return() → logs "cleanup!"
  console.log(n); // 0 1 2
}

P11·03

Custom Iterators — Advanced Patterns

1️⃣

Concept Explanation

Beyond basic iteration, custom iterators unlock lazy evaluation, infinite sequences, iterator composition (map, filter, take), and early-exit cleanup. These patterns are the same ones that power native iterators like String, Map, and Array entries.

Simple Explanation

A custom iterator is like a lazy pipeline: values are computed only when asked for, not all upfront. This means you can represent an infinite series or a huge dataset without blowing up memory.

MDN-Level Explanation

Iterator composition is achieved by chaining iterators: a map iterator wraps an inner iterator and transforms each value; a filter iterator wraps one and skips values not matching a predicate. Because each wrapper calls inner.next() lazily, no intermediate array is created. This is the model behind Array.prototype[Symbol.iterator] pipeline helpers (Stage 3 Iterator Helpers proposal) and is already available natively in some environments.

A stateful iterator may need cleanup (open file handles, event listeners, AbortController). Implement return(value) on the iterator to run cleanup when the consumer exits early.

Mental Model

Think of iterators as Unix pipes: each stage reads one value at a time from the previous stage and passes one value at a time to the next. Nothing buffers; the data flows on demand.

2️⃣

Code Examples

Basic — lazy map & filter iterators

  for (const v of iterable) yield fn(v);
}

  for (const v of iterable) if (pred(v)) yield v;
}
function  take(iterable, n) {
  const result = [];
  for (const v of iterable) { result.push(v); if (result.length === n) break; }
  return result;
}

  5
);

Intermediate — iterator with cleanup (return())

function linesIterator(text) {
  const lines = text.split('\n');
  let i = 0;
  let closed = false;
  return {
    [Symbol.iterator]() { return this; },
    next() {
      if (closed || i >= lines.length) return { done: true, value: undefined };
      return { value: lines[i++], done: false };
    },
    return() {
      closed = true;
      console.log('iterator closed early'); // e.g. close file handle here
      return { done: true, value: undefined };
    }
  };
}

for (const line of linesIterator('a\nb\nc\nd')) {
  console.log(line);
  if (line === 'b') break; // triggers return()
}

  return {

      let cursor = null;
      do {
        const { items, next } = await fetchPage(cursor);
        for (const item of items) yield item;
        cursor = next;
      } while (cursor);
    }
  };
}

P11·04

Generators

1️⃣

Concept Explanation

A generator function (function*) is a special function that can pause its own execution and resume it later. Each pause point is a yield expression. Calling a generator function does not run its body — it returns a generator object which is both an iterator and an iterable. Execution advances only when you call .next().

Simple Explanation

A generator is a function that can hit a "pause" button mid-execution (yield) and hand a value to whoever called it. When the caller asks for the next value, the function "unpauses" exactly where it stopped.

MDN-Level Explanation

A generator function is defined with function*. Calling it returns a Generator object — a closed-over activation record. Each call to generator.next(val):

  1. Resumes execution from the last yield (or from the start on the first call).
  2. The optional val argument becomes the result of the yield expression that paused the generator.
  3. Execution continues until the next yield, a return, or the end of the function.
  4. Returns { value: yieldedValue, done: false } or { value: returnValue, done: true }.

The generator object satisfies both the iterable protocol ([Symbol.iterator]() { return this; }) and the iterator protocol (next()), making it usable anywhere iterables are accepted.

Internal Working

The JS engine represents a suspended generator as a saved activation record: the program counter (where to resume), the local variable bindings, and the scope chain are all frozen on the heap. When .next() is called the engine restores this activation record and continues execution. This is fundamentally different from a regular function call which creates a brand-new frame and discards it on return. Generators are the language-level building block that async/await is compiled into by transpilers.

Mental Model

A generator is a function with a bookmark. yield places the bookmark and hands a value out; next() opens the book at the bookmark and continues reading.

2️⃣

Code Examples

Basic — simple sequence generator

function* count() {
  console.log('start');
  yield 1;
  console.log('after 1');
  yield 2;
  console.log('after 2');

}

const gen = count();       // body NOT yet executed
gen.next(); // logs "start" → { value: 1, done: false }
gen.next(); // logs "after 1" → { value: 2, done: false }

P11·05

Advanced Generator Features

1️⃣

Concept Explanation

Beyond basic yield, generators expose three advanced capabilities: yield* for delegation, .throw() for injecting errors, and .return() for forced termination. Together with async function* (async generators) these features enable sophisticated control-flow patterns like two-way communication pipelines, coroutines, and async data streaming.

yield* — Generator Delegation

yield* expr delegates to another iterable. It yields each value from expr as if those yields were written inline. When the inner iterable is a generator, its return value becomes the result of the yield* expression. This is how you compose generators without manually forwarding values.

.throw(err) — Injecting Errors

gen.throw(err) resumes the suspended generator and causes the paused yield to throw err. If the generator catches it, execution continues and throw() returns the next yielded value. If it doesn't, the error propagates to the caller. This is how async/await surfaces rejected promises inside await expressions.

async function* — Async Generators

An async generator is a generator that can await inside its body. It returns an async iterator — an object with an async next() that returns a Promise resolving to { value, done }. Consumed with for await...of. Perfect for streaming data sources: paginated APIs, WebSocket messages, file streams.

2️⃣

Code Examples

yield* delegation

  yield 'before';

  console.log('inner returned:', result);
  yield 'after';
}

P12·01

Event Loop, Call Stack & Task Queues

1️⃣

Concept Explanation

JavaScript is single-threaded — it can only do one thing at a time. Yet it handles timers, network requests, and user events concurrently without blocking. The event loop is the mechanism that makes this possible.

Simple Explanation

Imagine a chef (JS engine) who can only cook one dish at a time. When an order needs 20 minutes in the oven, the chef sets a timer and moves on. When the timer rings, the kitchen assistant (event loop) puts that order back in front of the chef so he can finish it. The chef never waits — he keeps cooking.

MDN-Level Explanation

The JavaScript runtime consists of several cooperating components:

  • Call Stack — a LIFO stack of execution contexts. When a function is called, a frame is pushed; when it returns, the frame is popped. Only one frame executes at a time.
  • Heap — unstructured memory region where objects are allocated.
  • Web APIs / Node APIs — browser/runtime-provided APIs (setTimeout, fetch, DOM events) that run outside the JS thread.
  • Macrotask Queue (Task Queue) — a FIFO queue of callbacks waiting to run. Each iteration of the event loop picks exactly one macrotask.
  • Microtask Queue — a higher-priority FIFO queue. After every task (and after each macrotask), all pending microtasks are drained before the next macrotask runs. Promise callbacks (.then, .catch, .finally) and queueMicrotask() schedule microtasks.
  • Event Loop — continually checks: if the call stack is empty, drain microtasks, then run the next macrotask, repeat.

Internal Working — Step by Step

The event loop algorithm (simplified from the HTML spec):

  1. Pick the oldest task from the macrotask queue and push it onto the call stack.
  2. Execute it until the stack is empty (all sync code runs).
  3. Drain the microtask queue: run every pending microtask in order; if a microtask enqueues more microtasks, those run too — before any macrotask.
  4. Optionally render (browser only — paint/layout if a frame is due).
  5. Go to step 1.

Key insight: microtasks starve the renderer and the next macrotask. A microtask that infinitely enqueues more microtasks will freeze the page forever.

Mental Model

Think of two waiting rooms at a doctor's office:
VIP waiting room (microtasks) — patients here are seen between every appointment, no matter how many there are.
General waiting room (macrotasks) — one patient is seen per appointment slot, but only after the VIP room is empty.
The doctor (JS engine) only treats one patient at a time, and VIP always goes first.

Asynchronous Programming with Callbacks

1️⃣

Concept Explanation

Before Promises, the only way to do async work in JavaScript was to pass a callback — a function you hand to another function, saying "call this when you're done." It was the original async pattern and still underlies everything, including Promises and async/await.

Simple Explanation

Ordering food at a restaurant: you give the waiter your order and your phone number. You go about your day. When the food is ready, the kitchen calls you. Your phone number is the callback — a way to reach you later.

MDN-Level Explanation

A callback is any function passed as an argument to another function with the expectation that the receiving function will invoke it at some later point. In Node.js and older browser APIs, the error-first (Node.js) convention standardises this: the first parameter of every callback is an Error object (or null if successful), followed by any result values. This lets a single callback handle both success and failure paths.

Internal Working

When you call setTimeout(cb, 1000):

  1. The runtime registers the timer in the Web API environment (outside JS thread).
  2. Your sync code continues — the call stack does not wait.
  3. After ~1000 ms the Web API pushes cb onto the macrotask queue.
  4. Once the call stack is empty and microtasks are drained, the event loop picks cb and executes it.

The callback itself is a plain function — it gets its own execution context, its own this, and closes over the variables in scope when it was defined (closure).

Mental Model — Callback Hell

When async operations depend on each other, callbacks nest. The resulting "pyramid of doom" grows to the right with each layer, making error handling a chore — you must handle errors at every level independently. This was the primary motivation for Promises.

P12·03

Promises

1️⃣

Concept Explanation

A Promise is an object representing the eventual completion (or failure) of an asynchronous operation and its resulting value. Introduced in ES2015, Promises replaced callback-based patterns and are the foundation of async/await.

Simple Explanation

A Promise is like a receipt you get when you order at a counter. You don't have the food yet, but you hold a token that says "food is coming." When the food is ready (fulfilled) you exchange the receipt for the food. If the kitchen burns down (rejected) the receipt is cancelled and you're told why.

MDN-Level Explanation

A Promise is in one of three mutually exclusive states:

  • Pending — initial state; neither fulfilled nor rejected.
  • Fulfilled — the operation completed successfully; the Promise has a value.
  • Rejected — the operation failed; the Promise has a reason (usually an Error).

Once settled (fulfilled or rejected) a Promise is immutable — its state and value can never change. Callbacks registered with .then(onFulfilled, onRejected) are always invoked asynchronously as microtasks, even if the Promise is already settled when .then() is called.

Internal Working

new Promise(executor) immediately calls executor(resolve, reject). The executor is synchronous. When you call resolve(value):

  1. The Promise transitions from pending → fulfilled and stores value.
  2. All registered .then(onFulfilled) handlers are enqueued as microtasks in registration order.

.then() always returns a new Promise. That child Promise resolves with whatever the handler returns (or rejects if the handler throws). This is what makes chaining work: each .then() produces a Promise that the next .then() waits for.

If onFulfilled returns a Promise (or any thenable), the engine assimilates it — the child Promise adopts that returned Promise's eventual state instead of resolving immediately with it.

Mental Model

A Promise is a single-use event emitter for one async result. A .then() chain is a pipeline: each stage transforms the value or handles an error before passing the result downstream. An unhandled rejection propagates down the chain looking for a .catch() — like an exception bubbling up a call stack.

Async / Await

1️⃣

Concept Explanation

async/await, introduced in ES2017, is syntactic sugar over Promises. It lets you write asynchronous code that looks and reads like synchronous code, without changing the underlying Promise mechanics at all.

Simple Explanation

Instead of .then().then().catch() chains, you write your async code top-to-bottom as if it were synchronous. The keyword await says "pause here, wait for this Promise to settle, then continue with the value." The keyword async on the function signals that it will always return a Promise.

MDN-Level Explanation

async function — declares an asynchronous function. It always returns a Promise. If the function body returns a non-Promise value v, the returned Promise resolves with v. If it throws, the returned Promise rejects with the thrown value.

await — valid only inside an async function (or at the top level of an ES module). It takes a value (typically a Promise) and suspends execution of the containing async function until the awaited Promise settles. The suspension point yields control back to the event loop — other code can run. When the Promise settles, execution resumes after the await expression, with the resolved value as the result (or a thrown exception if it rejected).

Internal Working

Under the hood, a transpiler (or the engine natively) transforms an async function into a state machine backed by a generator. Each await point becomes a yield, and the engine's Promise machinery resumes the generator via microtasks when the awaited Promise settles. Concretely:

  1. When an async function is called it begins executing synchronously.
  2. When it hits an await expr, it evaluates expr (getting a Promise), then suspends — returning an implicit "pending" Promise to the caller.
  3. The awaited Promise, once settled, enqueues a microtask that resumes the suspended function.
  4. Execution continues from the await point with the resolved value.

Because resumption is always via a microtask, await never blocks the call stack or starves other macrotasks.

Mental Model

async/await is like writing a recipe synchronously even though each step takes real time. "Bake for 30 minutes — await. Check if done. Frost the cake." The chef (your function) is paused (suspended) while the cake bakes, freeing up the kitchen (event loop) for other work.

Asynchronous Iteration

1️⃣

Concept Explanation

Synchronous iterators produce values immediately. But what if each value comes from the network, a file, or a database? Async iteration (ES2018) extends the iterator protocol to support values that arrive asynchronously — each next() call returns a Promise instead of a plain result object.

Simple Explanation

A synchronous iterator is like a vending machine — press the button, instantly get the snack. An async iterator is like a restaurant ordering system — press the button, get a receipt (Promise), and the snack arrives later. for await...of automates the "wait for each snack" loop.

MDN-Level Explanation

Async Iterator Protocol: an object is an async iterator if it has a next() method that returns a Promise resolving to { value, done }. An object is an async iterable if it has a [Symbol.asyncIterator]() method returning an async iterator.

for await...of — works only inside async functions (or top-level modules). For each iteration it calls iterator.next(), awaits the returned Promise, then assigns result.value to the loop variable. It stops when done is true.

Async Generators (async function*) — combine async functions and generators. yield produces a value (wrapped in a resolving Promise) and await can be used anywhere in the body. An async generator automatically satisfies the async iterable protocol.

Internal Working

When the engine processes for await (const x of asyncIterable):

  1. Calls asyncIterable[Symbol.asyncIterator]() to get an async iterator.
  2. Calls iterator.next() — gets a Promise.
  3. Awaits that Promise (suspends the async function, yields control to event loop).
  4. When the Promise resolves to { value, done }: assigns value to the loop variable, runs the loop body.
  5. If done === true, exits the loop.
  6. If the loop exits early (break, return, throw), calls iterator.return() — which returns a Promise — and awaits it before fully exiting.

An async generator is a state machine: each yield suspends and stashes the value, each await suspends and waits for a Promise. The engine queues resumptions as microtasks.

Mental Model

Think of a live news feed: each article is delivered asynchronously. for await...of is the journalist sitting at a terminal saying "give me the next article, wait for it to arrive, read it, then ask for the next one." An async generator is the API producing that feed — it can await network calls internally and yield each result as it arrives.

Object Extensibility

1️⃣

Concept Explanation

Simple Explanation

JavaScript lets you lock down objects at three levels of severity — from just blocking new properties, all the way to making everything completely read-only.

MDN-Level Explanation

Every object has an internal [[Extensible]] slot (defaults true). Setting it to false prevents adding new properties. JavaScript provides three escalating levels:

MethodAdd new?Delete?Change values?Change descriptors?
Object.preventExtensions()❌✅✅✅
Object.seal()❌❌✅❌
Object.freeze()❌❌❌❌

Inspect status with: Object.isExtensible(), Object.isSealed(), Object.isFrozen(). All three are shallow — nested objects remain mutable.

Internal Working

Object.freeze() internally calls Object.seal(), which calls Object.preventExtensions(), then marks all data properties writable: false. V8 can exploit frozen objects as fixed-shape structs for property access optimization. All lockdown operations are one-way — there is no unlock.

Mental Model

Three levels of a locked room
preventExtensions — rearrange and remove furniture, but you can't bring in new items.
seal — furniture is bolted down (can't remove), but you can repaint it (change values).
freeze — the entire room is encased in glass. Nothing changes at all.

The Prototype Attribute

1️⃣

Concept Explanation

Simple Explanation

Every JavaScript object has a hidden link to another object — its prototype. When you access a missing property, JS follows this link upward. You can read and change this link at runtime using dedicated APIs.

MDN-Level Explanation

The prototype is stored in the internal [[Prototype]] slot. The standard API for manipulating it:

OperationAPI
Get prototypeObject.getPrototypeOf(obj)
Set prototype (post-creation)Object.setPrototypeOf(obj, proto)
Create with prototypeObject.create(proto)
Legacy (avoid)obj.__proto__
Check in chainproto.isPrototypeOf(obj)

Every function has a .prototype property. When called with new, the instance's [[Prototype]] is set to that function's .prototype object.

Internal Working

Property lookup is a chain walk: V8 checks own properties first, then follows [[Prototype]] repeatedly until null. Calling Object.setPrototypeOf() after creation forces V8 to de-optimize — it breaks the "hidden class" shape assumption. Always prefer Object.create(proto) at construction time for hot-path objects.

Mental Model

Think of the prototype chain as a stack of recipe cards
Your object is the top card. If a property is missing, look at the card below (the prototype), then the card below that, until you reach a blank card (null). Each card can shadow values from cards below it.
2️⃣

P13·04

Symbols

1️⃣

Concept Explanation

Simple Explanation

A Symbol is a guaranteed-unique primitive value that can be used as an object property key. No two symbols are ever equal — even if created with the same description string.

MDN-Level Explanation

Symbol is the seventh primitive type in JavaScript (ES6+). Unlike strings, each call to Symbol() returns a unique value that cannot be implicitly coerced to a string. Symbol-keyed properties are non-enumerable by default — they don't appear in for...in or Object.keys() — making them ideal for hidden metadata or protocol hooks without naming conflicts.

Internal Working

Symbols are allocated on the heap but behave like primitives. The engine guarantees uniqueness at the allocation level (not by name comparison). Symbol.for(key) uses a global registry shared across realms/iframes. Symbol() creates a realm-local unique value not in any registry. V8 stores symbol-keyed properties in a separate slot from string-keyed properties within the object's hidden class.

Mental Model

Think of a Symbol as a one-of-a-kind physical key
Even if two keys look identical (same description), they open different locks. Symbol.for() is a key-cutting shop — ask for the same name twice and you get the exact same key. Symbol() cuts a brand-new unique key every time, guaranteed to match nothing else.

Template Tags

1️⃣

Concept Explanation

Simple Explanation

A tagged template literal lets you process a template string with a custom function — the "tag". Instead of producing a plain string, you intercept the static parts and the interpolated values and can return anything you want.

MDN-Level Explanation

When a template literal is preceded by an expression (the tag), the tag function is called with an array of static string parts (strings) and the evaluated interpolated values as rest parameters (...values). It can return any value — a string, an object, a DOM node.

tag`Hello ${name}, you are ${age}` is equivalent to:
tag(["Hello ", ", you are ", ""], name, age)

strings.raw contains the raw (unescaped) string parts — \n stays as \n instead of becoming a newline.

Internal Working

The parser splits the template at compile time into alternating static chunks and dynamic expressions. The static chunks are stored as a frozen array that is reused across all calls to the same template literal — the same array reference every time. Tag functions can use this for caching (e.g., compile the template once). Values are re-evaluated on each call.

Mental Model

Think of a tag as a printf formatter under your control
The template gives you alternating fixed bread and variable fillings. The tag function decides how to assemble the sandwich — sanitise the fillings, wrap them in HTML, build a SQL query, or return an entirely different type.
2️⃣

// Real-world 1: HTML escaping to prevent XSS
function html(strings, ...values) {
  const escape = (s) => String(s)
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;");
  return strings.reduce((acc, str, i) =>
    acc + str + (i < values.length ? escape(values[i]) : ""), ""
  );
}

// Real-world 2: SQL parameterised query builder

    acc + str + (i < values.length ? `$${i + 1}` : ""), ""
  );
  return { query, params: values };
}

1️⃣

Concept Explanation

Simple Explanation

A Proxy wraps an object and lets you intercept and customise any fundamental operation on it — reading, writing, calling, deleting. You define what happens in a handler object whose methods are called traps.

MDN-Level Explanation

new Proxy(target, handler) creates a proxy for target. The handler is a plain object whose methods (traps) intercept operations. If a trap is not defined, the operation passes through to the target automatically. Traps must not violate the invariants of the target — e.g., a non-configurable property's true value must be returned by a get trap.

Every JavaScript operation on an object is defined in terms of internal methods like [[Get]], [[Set]], [[Call]]. A Proxy replaces these with trap calls. V8 implements proxies through a layer of indirection — there is a real performance cost (~2–5× slower than direct property access) so avoid proxies on hot paths.

Mental Model

A Proxy is a bodyguard for your object
Every request to read, write, or call goes through the bodyguard first. The bodyguard can allow it, block it, modify it, or log it. The caller has no idea a bodyguard is present — they see the same interface.
2️⃣

Code Examples

JavaScriptProxy Patterns
// 1. Logging proxy — observe all reads and writes
function createLogged(obj) {
  return new Proxy(obj, {
    get(target, prop) {
      console.log(`GET: ${String(prop)}`);
      return Reflect.get(target, prop);
    },
    set(target, prop, value) {
      console.log(`SET: ${String(prop)} = ${value}`);
      return Reflect.set(target, prop, value);
    }
  });
}
const user = createLogged({ name: "Vipin" });
user.name;      // logs: GET: name
user.age = 25; // logs: SET: age = 25

// 2. Validation proxy — enforce schema
const validator = new Proxy({}, {
  set(target, prop, value) {
    if (prop === "age" && (typeof value !== "number" || value < 0))
      throw new TypeError("age must be a non-negative number");
    return Reflect.set(target, prop, value);
  }
});
validator.age = 25;   // ✅
validator.age = "old"; // ❌ TypeError

// 3. Default-value proxy
function withDefault(obj, defaultValue) {
  return new Proxy(obj, {
    get(target, prop) {
      return prop in target ? Reflect.get(target, prop) : defaultValue;
    }
  });
}
const scores = withDefault({ alice: 95 }, 0);
scores.alice; // 95
scores.bob;   // 0 — default

// 4. Revocable proxy — temporary access
const { proxy, revoke } = Proxy.revocable({ secret: "token-xyz" }, {});
proxy.secret; // "token-xyz"
revoke();
proxy.secret; // ❌ TypeError: Cannot perform 'get' on a revoked proxy

// 5. has trap — hide private properties from 'in'
const hidden = new Proxy({ _private: 42, pub: "hello" }, {
  has(target, prop) {
    return prop.startsWith("_") ? false : Reflect.has(target, prop);
  }
});
"pub" in hidden;      // true
"_private" in hidden; // false

Common Mistakes

❌ Forgetting to return true from the set trap
JSWrong → Fixed
const p = new Proxy({}, {
  set(target, prop, value) {
    target[prop] = value; // ❌ returns undefined → TypeError in strict mode
  }
});
// ✅ Always return Reflect.set(…) which returns the boolean result
set(target, prop, value) {
  return Reflect.set(target, prop, value); // ✅
}
❌ Accessing the proxy inside a trap — causes infinite recursion
JSWrong → Fixed
get(target, prop) {
  return proxy[prop]; // ❌ calls the get trap again → stack overflow!
}
// ✅ Always read from target (or use Reflect) inside traps
get(target, prop) {
  return Reflect.get(target, prop); // ✅
}

Reflect API

1️⃣

Concept Explanation

Simple Explanation

Reflect is a built-in object that provides function equivalents of JavaScript's meta-operations — the same things you do with operators (in, delete, new). It is the canonical way to forward operations inside a Proxy trap.

MDN-Level Explanation

Reflect is not a constructor — you cannot new Reflect(). Its static methods mirror exactly the internal methods of JavaScript objects. Every Proxy trap has an exact Reflect counterpart with the same signature, making default-forwarding trivial and correct.

Reflect MethodEquivalent To
Reflect.get(obj, key, recv)obj[key]
Reflect.set(obj, key, val, recv)obj[key] = val
Reflect.has(obj, key)key in obj
Reflect.deleteProperty(obj, key)delete obj[key]
Reflect.apply(fn, thisArg, args)fn.apply(thisArg, args)
Reflect.construct(Cls, args, newTarget?)new Cls(...args)
Reflect.defineProperty(obj, k, d)Object.defineProperty()
Reflect.getPrototypeOf(obj)Object.getPrototypeOf()
Reflect.setPrototypeOf(obj, p)Object.setPrototypeOf()
Reflect.ownKeys(obj)All own string + symbol keys
Reflect.isExtensible(obj)Object.isExtensible()
Reflect.preventExtensions(obj)Object.preventExtensions()

Key Differences from Object.*

Unlike Object.defineProperty() which throws on failure, Reflect.defineProperty() returns false. All Reflect methods that can fail return a boolean — enabling control-flow checks instead of try/catch. Also, Reflect.apply(fn, thisArg, args) works even if fn.apply has been overridden or proxied.

Internal Working

Reflect methods map directly to V8's internal methods ([[Get]], [[Set]], etc.) — the very same methods Proxy traps intercept. They are the "default implementations" of those internal slots. Using Reflect.* inside traps ensures the correct receiver is forwarded (important for inherited getters/setters).

Mental Model

Reflect is the instruction manual for Proxy traps
Proxy is the telephone switchboard operator who intercepts calls. Reflect is the manual that says exactly what each call would normally do. Inside a trap, use Reflect to route the call through normally — possibly with logging or modifications applied first.

Code Examples

JavaScriptReflect API
// Reflect.get — respects receiver for inherited getters
const obj = {
  _x: 10,
  get x() { return this._x * 2; }
};
const extended = { _x: 5 };
Reflect.get(obj, "x", extended); // 10 — getter runs with extended as this

// Reflect.set — returns false on failure (no throw)
const frozen = Object.freeze({ a: 1 });
Reflect.set(frozen, "a", 99); // false — silently fails, no exception

// Reflect.apply — safe even if fn.apply is overridden
Reflect.apply(Math.max, null, [3, 1, 4, 1, 5]); // 5

// Reflect.construct — build instance with different new.target
class A { constructor(x) { this.x = x; } }
class B {}
const inst = Reflect.construct(A, [42], B);
inst.x;              // 42 — constructed with A's logic
inst instanceof B;  // true — prototype set to B.prototype

// Reflect.ownKeys — all own keys including non-enumerable and symbols
const sym = Symbol("s");
const o = { a: 1, [sym]: 2 };
Object.keys(o);     // ["a"] — no symbols
Reflect.ownKeys(o); // ["a", Symbol(s)] — everything

// Reflect.defineProperty — boolean return, no throw
const f = Object.freeze({});
try {
  Object.defineProperty(f, "x", { value: 1 }); // ❌ throws TypeError
} catch(e) {}
Reflect.defineProperty(f, "x", { value: 1 }); // false — no throw!

// Proxy + Reflect — canonical complete forwarding pattern
function createProxy(target) {
  return new Proxy(target, {
    get(t, p, r)         { return Reflect.get(t, p, r); },
    set(t, p, v, r)      { return Reflect.set(t, p, v, r); },
    has(t, p)            { return Reflect.has(t, p); },
    deleteProperty(t, p) { return Reflect.deleteProperty(t, p); }
  });
}

Common Mistakes

❌ Using Object.defineProperty inside a Proxy trap (throws instead of returning false)
JSWrong → Fixed
defineProperty(target, key, desc) {
  Object.defineProperty(target, key, desc); // ❌ throws on failure, no return value
}
// ✅ Reflect returns false on failure — no throw, proper control-flow
defineProperty(target, key, desc) {
  return Reflect.defineProperty(target, key, desc); // ✅ boolean
}
❌ Skipping the receiver in Reflect.get / Reflect.set

You must pass the receiver argument: Reflect.get(target, prop, receiver). Without it, this inside a getter runs against target, not the proxy — breaking inherited getters that depend on the correct this.
