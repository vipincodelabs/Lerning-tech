01

Variables — var, let, const

1️⃣

Concept Explanation

A variable is a named container that holds a value in memory. JavaScript has three keywords — var, let, const — each with different rules for scope, hoisting, and reassignment.

KeywordScopeHoistedRe-declarableReassignableTDZ
varFunction✅ as undefined✅ Yes✅ Yes❌ No
letBlock✅ uninitialized❌ No✅ Yes✅ Yes
constBlock✅ uninitialized❌ No❌ No✅ Yes

Internal Working — Two-Phase Execution

Phase 1 — Compilation (before any code runs):
┌──────────────────────────────────────────┐
│ var x → undefined │ ← hoisted & initialized
│ let y → <TDZ — not initialized> │ ← hoisted, NOT initialized
│ const z → <TDZ — not initialized> │ ← hoisted, NOT initialized
└──────────────────────────────────────────┘

Phase 2 — Execution (line by line):
values assigned when engine reaches those lines
⚠️ var leaks out of blocks

var does NOT respect {} blocks. It leaks into the nearest function scope. This is the #1 source of var bugs.

⚠️ const ≠ immutable

const prevents the binding from being reassigned. The object's properties can still change.

🧠 Mental Model — Labeled Boxes

var — A box in the whole room (function). Exists before you fill it. Works anywhere in the room.

let — A box on one shelf (block). Cannot leave the shelf. You can refill it.

const — A permanently labeled box. You can't re-label it, but you can rearrange what's inside (for objects/arrays).

2️⃣

Code Examples

JavaScriptBasic
// var — re-declarable, function scoped
var city = "Delhi";
var city = "Mumbai"; // ✅ no error

// let — block scoped, reassignable
let score = 0;
score = 100; // ✅

// const — binding cannot be reassigned
const PI = 3.14159;
// PI = 3; // ❌ TypeError
JavaScriptReal-World — Loop Bug
// ❌ var: all callbacks share same i
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log("var:", i), 100);
}
// Output: var: 3, var: 3, var: 3

// ✅ let: each iteration gets its own scoped i
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log("let:", i), 100);
}
// Output: let: 0, let: 1, let: 2

Common Mistakes

02

Primitive vs Reference Types

1️⃣

Concept Explanation

Every JavaScript value is either a primitive (stored by value) or a reference type (stored by reference). The difference determines how values are copied and compared.

CategoryTypesStored InCopied ByMutable?
Primitivestring number bigint boolean undefined null symbolStackValue❌ No
ReferenceObject Array Function Date Map SetHeap (pointer on stack)Reference✅ Yes

Internal Working — Stack vs Heap

STACK (fast, fixed-size) HEAP (large, dynamic)
┌──────────────────────┐ ┌───────────────────────────┐
│ a = 10 │ │ │
│ b = 10 ← full copy │ │ { name: "Vipin" } ←─────┼── user1
│ │ │ (same object) ←─────┼── user2
│ user1 = [ref:0x001] ─┼──────────►│ [address: 0x001] │
│ user2 = [ref:0x001] ─┼──────────►│ │
└──────────────────────┘ └───────────────────────────┘

Changing b does NOT affect a (different stack slots)
Changing user2.name DOES affect user1 (same heap object)
🧠 Mental Model

Primitive = photocopy. Give someone a copy — they mark it up, your original is fine.

Reference = Google Doc link. Share the link — they edit it, and you both see the changes.

2️⃣

Code Examples

JavaScriptBasic → Advanced
// PRIMITIVES — copied by value
let a = 10;
let b = a;        // b gets a full COPY of 10
b = 99;
console.log(a);   // 10 — unchanged
console.log(b);   // 99

// REFERENCE — copied by pointer
let user1 = { name: "Vipin" };
let user2 = user1;           // same object in memory
user2.name = "Singh";
console.log(user1.name);     // "Singh" — user1 affected!

// COMPARISON
console.log({} === {});      // false — different objects
console.log([] === []);      // false — different arrays
console.log("hi" === "hi"); // true  — same primitive value

// SHALLOW COPY — spread operator
const original = { score: 100 };
const clone = { ...original };
clone.score = 0;
console.log(original.score);  // 100 — untouched ✅

// DEEP COPY — JSON (flat objects only)
const deep = { user: { prefs: { theme: "dark" } } };
const copy = JSON.parse(JSON.stringify(deep));
copy.user.prefs.theme = "light";
console.log(deep.user.prefs.theme); // "dark" — untouched ✅
// ⚠️ JSON.stringify drops: functions, undefined, Dates, Symbols

// REAL-WORLD: Settings draft isolation
const liveSettings = { theme: "dark", fontSize: 16 };
let draft = { ...liveSettings };     // isolated copy
draft.theme = "light";              // editing draft doesn't affect live
// On save:
// Object.assign(liveSettings, draft);

📌 When does undefined appear automatically?

🧠 Mental Model

undefined = An empty mailbox. Nothing was ever put there.

null = An empty mailbox with a sign saying "Empty on purpose."

2️⃣

Code Examples

JavaScriptAll Cases

console.log(x);                          // undefined

const obj = { a: 1 };
console.log(obj.b);                      // undefined — no property

// null — set by developer
let currentUser = null;                  // intentional: no user logged in
currentUser = { name: "Vipin" };        // now assigned

// The equality trap
console.log(null == undefined);         // true  — loose eq
console.log(null === undefined);        // false — strict eq
console.log(typeof null);               // "object" ← JS bug!
console.log(typeof undefined);          // "undefined"

// Safe check for BOTH null AND undefined
function process(value) {
  if (value == null) {                   // catches null AND undefined
    console.log("No value provided");
    return;
  }
  console.log(value);
}
process(null);       // "No value provided"
process(undefined);  // "No value provided"
process(42);         // 42

// JSON behaviour difference
console.log(JSON.stringify({ a: undefined, b: null }));
// {"b":null}  — undefined is omitted, null is kept

The Global Object

1️⃣

Concept Explanation

The global object is the top-level container in a JavaScript runtime. Every built-in (Math, JSON, setTimeout) and every var declared at the top level lives on it.

EnvironmentGlobal ObjectUniversal (ES2020+)
BrowserwindowglobalThis
Node.jsglobal
Web Workerself
📌 Key Behavior
  • var declarations at top level → become properties of the global object
  • let / const at top level → do NOT become global object properties
  • All built-ins (Math, parseInt, setTimeout) live on the global object
JavaScriptBrowser
var appName = "MyApp";
console.log(window.appName);         // "MyApp" — on window!

let version = "1.0";
console.log(window.version);         // undefined — not on window

// globalThis works everywhere
console.log(globalThis.Math.PI);     // 3.14...
console.log(globalThis.parseInt("42")); // 42

// Environment detection
const isBrowser = typeof window !== "undefined";
const isNode = typeof process !== "undefined" && process.versions?.node;
console.log(isBrowser ? "Browser" : "Node");

function getAge(input) {

  return parsed;
}

Common Mistakes

❌ Using == instead of ===

Loose equality triggers coercion. userInput == 0 is true for "", false, null, "0". Always use ===.

❌ Assuming [] and {} are falsy

❌ parseInt without radix

parseInt("010") could be 8 (octal) in some environments. Always: parseInt("010", 10).

JavaScriptPrecedence in Action
// Arithmetic
2 + 3 * 4       // 14 — * before +
(2 + 3) * 4     // 20 — () overrides

// Right-to-left: exponentiation
2 ** 3 ** 2     // 512 — 2**(3**2) = 2**9, NOT (2**3)**2 = 64!

// && has higher precedence than ||
true || false && false  // true — evaluated as: true || (false && false)

// The + concatenation trap
"1" + 2 + 3    // "123" — "1"+"2" = "12", "12"+"3" = "123"
1 + 2 + "3"    // "33"  — 1+2 = 3, 3+"3" = "33"

// typeof binds tighter than +
typeof 1 + "2"  // "number2" — (typeof 1) + "2"
typeof (1 + "2") // "string"  — typeof("12")

// Assignment is right-to-left
let a, b, c;
a = b = c = 5;  // c=5, b=5, a=5 ✅

// Real-world: always use parens for complex logic
// ❌ Ambiguous

// ✅ Explicit intent
const access2 = (isLoggedIn && hasPermission) || isAdmin;

null == 0            // false — null only == null/undefined

Nullish Coalescing ?? and Optional Chaining ?.

JavaScriptES2020 Operators
// ?? — returns right side ONLY if left is null or undefined
0 || "default"       // "default" — 0 is falsy, || triggers
0 ?? "default"       // 0         — 0 is not null/undefined, ?? keeps it
null ?? "fallback"  // "fallback"
"" ?? "fallback"    // ""         — empty string kept by ??

// ?. — optional chaining: safe nested access
const user = null;
// user.name              // ❌ TypeError
user?.name               // ✅ undefined — no throw

const config = { db: { host: "localhost" } };
config?.db?.host         // "localhost"
config?.cache?.host      // undefined — safe
config?.db?.connect?.()  // undefined — optional method call

// Combine: safe nested access with fallback
const city = response?.data?.user?.address?.city ?? "Unknown";

JavaScript||= &&= ??=
let a = null;
a ||= "default";   // a = a || "default" → "default"
a &&= "updated";   // a = a && "updated" → "updated" (a was truthy)

let b = null;
b ??= "fallback";  // b = b ?? "fallback" → "fallback"

P2·04

Short-Circuiting

1️⃣

Concept Explanation

Short-circuiting means JavaScript stops evaluating an expression as soon as the result is determined. The right side may never execute.

&& → stops at FIRST FALSY → returns that falsy value (or last if all truthy)
|| → stops at FIRST TRUTHY → returns that truthy value (or last if all falsy)
?? → stops at FIRST non-null/undefined → returns it

Return values are NOT necessarily boolean — they return the actual operand value
JavaScriptAll Patterns
// Side effect test — right side may never run
let count = 0;
false && count++;   // count stays 0 — right side NOT evaluated
true && count++;    // count becomes 1 — right side IS evaluated

// Pattern 1: Conditional execution
user.isAdmin && showAdminPanel(); // only runs if isAdmin is truthy

// Pattern 2: Default value with || (triggers on ANY falsy)
function greet(name) {
  name = name || "Guest";    // "" triggers this — careful!
  return `Hello, ${name}!`;
}

// Pattern 3: Default value with ?? (only null/undefined)
function greet2(name) {
  name = name ?? "Guest";    // "" is preserved ✅
  return `Hello, ${name}!`;
}

// Pattern 4: Lazy initialization / memoization
let cache = null;
function getData() {
  return cache || (cache = fetchFromServer()); // fetches once, then cached
}

// Pattern 5: Conditional rendering (React-style)
const isLoggedIn = true;
const ui = isLoggedIn && '<div>Dashboard</div>'; // shows div

// ❌ Danger: number as condition renders "0" in React
const count = 0;
const bad = count && '<span>items</span>'; // renders 0, not nothing!
// ✅ Fix: use boolean condition
const good = count > 0 && '<span>items</span>';
3️⃣

Common Mistakes

❌ Number condition in JSX/templates renders "0"

When count is 0, count && <span> renders the number 0 to the DOM, not nothing. Always use count > 0 && or a ternary.

❌ Side effects inside short-circuit — hard to debug

false && (x = 10) — the assignment NEVER runs. If you depend on a side effect, don't hide it inside a short-circuit expression.
