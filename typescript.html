01

What is TypeScript & How it Differs from JavaScript

1

📘 Definition

TypeScript is a strongly typed programming language that builds on JavaScript. It adds a static type system — meaning types are checked at compile time, before your code ever runs in a browser or Node.js.

TypeScript is a superset of JavaScript: every valid JavaScript file is also valid TypeScript. You add types on top; TypeScript compiles away to plain JavaScript.

FeatureJavaScriptTypeScript
TypingDynamic (at runtime)Static (at compile time)
Error detectionRuntime errorsCompile-time errors
Runs in browser✅ Directly❌ Must compile first
IDE supportLimited autocompleteFull IntelliSense + type hints
Refactoring safetyRiskySafe (types catch regressions)
File extension.js.ts / .tsx
2

📖 Official Insight

📖 TypeScript Docs say…

"TypeScript is a typed superset of JavaScript that compiles to plain JavaScript." — The TypeScript team describes it as a tool that "adds optional static typing" to JavaScript. The compiler is called tsc (TypeScript Compiler). The type annotations you write are erased at compile time — they have zero runtime cost.

3

🧠 Core Concept — The TypeScript Pipeline

Your code (.ts): const age: number = 25;
↓
TypeScript Compiler (tsc):
┌─────────────────────────────────────────┐
│ 1. Parse → AST (Abstract Syntax Tree) │
│ 2. Type-check → errors reported HERE │
│ 3. Emit → strip types, output .js │
└─────────────────────────────────────────┘
↓
Output (.js): const age = 25; ← types erased

This is the key insight: TypeScript types exist only during development. At runtime, JavaScript has no knowledge of them. TypeScript is purely a developer tool.

🧠 Mental Model

JavaScript is like writing code without spell-check. Errors show up only when you run it — often in production.

TypeScript is like writing with a smart editor that underlines mistakes before you even save the file. The spell-check disappears once the document is printed (compiled), but it saved you from errors.

4

⚙️ Syntax — Your First TypeScript

TypeScriptBasic
// JavaScript — no type annotation
let username = "Vipin";
username = 42; // ⚠️ No error in JS — silently wrong

// TypeScript — explicit type annotation
let username: string = "Vipin";
// username = 42; ❌ Type 'number' is not assignable to type 'string'

// Type annotation syntax: variableName: Type
let age: number = 25;
let isLoggedIn: boolean = true;
let greeting: string = "Hello";
5

🔥 Real-World Examples

TypeScriptWhy JS alone fails at scale
// JavaScript — no guardrails
function createUser(name, age, role) {
  return { name, age, role };
}

// What if a caller passes wrong args? JS has NO idea:
createUser(42, "admin", true); // silently accepted ❌

// ─────────────────────────────────────────────────
// TypeScript — contract enforced by the compiler
function createUser(name: string, age: number, role: string) {
  return { name, age, role };
}

createUser(42, "admin", true);
// ❌ Argument of type 'number' is not assignable to parameter of type 'string'
// Caught at compile time, not in production ✅
TypeScript + ReactProps typing
// Without TypeScript — any prop can be passed, no warnings
function UserCard({ name, age }) {
  return <div>{name} — {age}</div>;
}

// With TypeScript — caller knows exactly what to pass
type UserCardProps = {
  name: string;
  age: number;
};

function UserCard({ name, age }: UserCardProps) {
  return <div>{name} — {age}</div>;
}

// ❌ <UserCard name={42} age="twenty" />  — caught at compile time
// ✅ <UserCard name="Vipin" age={25} />   — correct
6

⚠️ Pitfalls

❌ Using `any` everywhere defeats TypeScript's purpose

any turns off type checking for that value. New developers often reach for any to silence errors — this gives you the worst of both worlds: no safety AND false confidence. Prefer unknown if you truly don't know the type.

❌ TypeScript ≠ runtime validation

TypeScript types are erased at runtime. If an API returns { age: "25" } instead of { age: 25 }, TypeScript won't catch it. Use Zod or io-ts for runtime validation of external data.

02

Type System Basics

1

📘 Definition

TypeScript's type system is the set of rules that determines which values can be assigned to which variables. It works by statically analyzing your code before it runs — this is called static analysis.

The type system has two modes: structural typing (the default, shape-based) and nominal typing (name-based, not default in TS). Understanding structural typing is fundamental to understanding why TypeScript behaves the way it does.

2

📖 Official Insight

📖 TypeScript Docs say…

TypeScript uses a structural type system. This means type compatibility is determined by the shape (structure) of the type, not its name. "If it walks like a duck and quacks like a duck, it's a duck." — This is often called duck typing in the TypeScript world.

3

🧠 Core Concept — Structural vs Nominal

TypeScriptStructural typing — shape is what matters
type Point2D = { x: number; y: number };
type Coordinates = { x: number; y: number };

const p: Point2D = { x: 1, y: 2 };
const c: Coordinates = p; // ✅ compatible — same shape!

// In a nominal system (Java/C#), this would fail.
// In TypeScript, if the SHAPE matches, it's compatible.

// Extra properties are OK when assigning (structural compatibility)
type Animal = { name: string };
type Dog = { name: string; breed: string };

const myDog: Dog = { name: "Rex", breed: "Husky" };
const animal: Animal = myDog; // ✅ Dog is assignable to Animal (has more, not less)
4

⚙️ Syntax — Type Annotations

TypeScriptAll annotation forms
// Variable annotation
let name: string;
const port: number = 3000;

// Function parameter + return type annotation
function greet(name: string): string {
  return `Hello, ${name}`;
}

// Object type annotation
let user: { name: string; age: number } = { name: "Vipin", age: 25 };

// Array annotation
let scores: number[] = [90, 85, 95];

// void — function that returns nothing
function logMessage(msg: string): void {
  console.log(msg);
}
5

🔥 Real-World — API Response Typing

TypeScriptTyped API response
// Define the shape of what your API returns
type ApiUser = {
  id: number;
  name: string;
  email: string;
  createdAt: string;
};

// TypeScript knows exactly what .name, .email are
async function fetchUser(id: number): Promise<ApiUser> {
  const res = await fetch(`/api/users/${id}`);
  return res.json() as ApiUser;
}

// Now the IDE gives you full autocomplete on the result
const user = await fetchUser(1);
console.log(user.name);  // ✅ TypeScript knows this is string
// console.log(user.xyz) // ❌ Property 'xyz' does not exist
6

⚠️ Pitfalls

❌ Excess property check at object literals — surprising behavior
TypeScriptPitfall
type User = { name: string };

// ❌ Fresh object literal triggers excess property check
const u: User = { name: "Vipin", age: 25 };
// Error: Object literal may only specify known properties

// ✅ Through a variable — no excess property check
const obj = { name: "Vipin", age: 25 };
const u2: User = obj; // fine — structural compatibility

TypeScript applies excess property checks only on fresh object literals assigned directly to a typed variable. This is intentional to catch typos.

03

Type Inference

1

📘 Definition

Type inference is TypeScript's ability to automatically determine the type of a value without you explicitly annotating it. You don't always need to write : string — TypeScript can figure it out from context.

2

📖 Official Insight

📖 TypeScript Docs say…

"In TypeScript, there are several places where type inference is used to provide type information when there is no explicit type annotation. For example, in this code: let x = 3; the type of x is inferred to be number." — TypeScript calls this contextual typing when inference flows from context (like event handlers).

3

🧠 Core Concept — How Inference Works

TypeScriptInference in action
// TypeScript infers 'string' from the initializer
let city = "Delhi";        // inferred: string
let count = 0;             // inferred: number
let active = true;        // inferred: boolean

// Inferred return type from function body
function add(a: number, b: number) {
  return a + b;            // return type inferred: number
}

// Array inference
const items = [1, 2, 3];  // inferred: number[]
const mixed = [1, "two"]; // inferred: (string | number)[]

// Contextual typing — TypeScript infers from context
const names = ["Vipin", "Singh"];
names.forEach(name => {
  console.log(name.toUpperCase()); // TS knows 'name' is string
});
4

⚙️ When to Annotate vs Infer

Rule of thumb:
• Let TypeScript infer when the type is obvious from the initial value.
• Annotate explicitly for: function parameters, function return types, variables declared without an initial value, and public API boundaries.

TypeScriptWhen to annotate
// ✅ Let inference work — obvious from initializer
const name = "Vipin";          // no annotation needed
const scores = [90, 85];       // inferred: number[]

// ✅ Annotate — declared without value
let userId: number;            // must annotate; no initializer
userId = 42;

// ✅ Annotate function parameters — TS can't infer these
function multiply(a: number, b: number): number {
  return a * b;
}

// ✅ Annotate return type — explicit contract, prevents mistakes
function getUsername(): string {
  return "Vipin";
}
5

🔥 Real-World — Inference in React useState

TypeScript + ReactuseState inference
import { useState } from 'react';

// TypeScript infers state type from initial value
const [count, setCount] = useState(0);     // count: number
const [name, setName] = useState("");      // name: string

// When type can't be inferred from initial value — annotate explicitly
const [user, setUser] = useState<User | null>(null);
// Without the generic, TS would infer: null — too narrow!
6

⚠️ Pitfalls

❌ Widened inference — `let` vs `const`
TypeScriptWidening
const status = "active";  // inferred: "active" (literal type)
let status2 = "active";  // inferred: string (widened!)

// 'const' infers the exact literal type because it can't change.
// 'let' widens to the broader type because it could change later.

2

📖 Official Insight — any vs unknown vs never

📖 TypeScript Docs say…

any is the escape hatch — it disables type checking entirely. unknown is the type-safe counterpart: you can assign anything to it, but you must narrow it before using it. never represents values that never occur — it appears in exhaustive checks and functions that always throw.

3

🧠 Core Concept

TypeScriptany vs unknown vs never
// any — no type safety, avoid in production code
let data: any = fetchSomething();
data.doAnything();   // ✅ no errors — but unsafe!

// unknown — must narrow before using
let result: unknown = fetchSomething();
// result.doAnything(); // ❌ Object is of type 'unknown'

if (typeof result === "string") {
  console.log(result.toUpperCase()); // ✅ now safe
}

// never — exhaustive checking
type Shape = "circle" | "square";

function getArea(shape: Shape) {
  switch (shape) {
    case "circle":  return 3.14;
    case "square": return 1;
    default:
      const _exhaustive: never = shape; // TS error if new shape added!
      throw new Error(`Unknown shape`);
  }
}
4

⚙️ Literal Types

TypeScriptLiteral types
// A literal type restricts a variable to exact values
let direction: "left" | "right" | "up" | "down";
direction = "left";   // ✅
// direction = "diagonal"; // ❌

type HttpMethod = "GET" | "POST" | "PUT" | "DELETE";
type StatusCode = 200 | 201 | 400 | 401 | 404 | 500;

function request(url: string, method: HttpMethod) {
  // method is guaranteed to be one of the four values
}
5

🔥 Real-World — API Error Typing with unknown

TypeScriptError handling with unknown
// Before TypeScript 4.0, catch errors were 'any'
// Now they're 'unknown' with useUnknownInCatchVariables

async function fetchData(url: string) {
  try {
    const res = await fetch(url);
    return res.json();
  } catch (error: unknown) {
    // Must narrow before accessing properties
    if (error instanceof Error) {
      console.error(error.message);  // ✅ safe
    } else {
      console.error("Unknown error", error);
    }
  }
}
6

⚠️ Pitfalls

❌ Using `any` instead of `unknown` for external data

When you receive data from an API, catch block, or JSON.parse(), use unknown not any. With unknown you must validate before using — this forces safe code. With any you skip validation and runtime errors can occur.

05

Functions & Objects in TypeScript

1

📘 Definition

Functions in TypeScript can be typed at every boundary: parameters, return type, and even callback shapes. Objects get typed with inline type annotations, type aliases, or interface declarations.

4

⚙️ Syntax — Function Typing Patterns

TypeScriptAll function forms
// 1. Named function with parameter + return types
function greet(name: string): string {
  return `Hello, ${name}`;
}

// 2. Arrow function
const double = (n: number): number => n * 2;

// 3. Optional parameter (use ?)
function createUser(name: string, role?: string) {
  return { name, role: role ?? "user" };
}

// 4. Default parameter
function paginate(page: number = 1, limit: number = 10) {
  return { page, limit };
}

// 5. Rest parameters
function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}

// 6. Function type as a type alias
type Transformer<T> = (value: T) => T;
const double: Transformer<number> = n => n * 2;
TypeScriptObject typing
// Inline object type
function printUser(user: { name: string; age: number }) {
  console.log(user.name, user.age);
}

// Named type alias (preferred for reuse)
type User = {
  id: number;
  name: string;
  email: string;
  role: "admin" | "user";
};

// Readonly — prevents mutation
type Config = {
  readonly apiUrl: string;
  readonly timeout: number;
};

const config: Config = { apiUrl: "/api", timeout: 5000 };
// config.apiUrl = "other"; // ❌ Cannot assign to 'apiUrl' — it's readonly
5

🔥 Real-World — Express.js Typed Middleware

TypeScript + ExpressTyped middleware
import { Request, Response, NextFunction } from 'express';

// Extend Request to carry typed user data
interface AuthRequest extends Request {
  user?: { id: number; role: string };
}

function authMiddleware(
  req: AuthRequest,
  res: Response,
  next: NextFunction
): void {
  const token = req.headers.authorization;
  if (!token) {
    res.status(401).json({ error: "Unauthorized" });
    return;
  }
  req.user = { id: 1, role: "admin" }; // typed ✅
  next();
}
6

⚠️ Pitfalls

❌ Missing return type on exported functions

Always annotate return types on exported functions (API boundaries). Inference is fine for internal code, but an explicit return type acts as a contract — if you accidentally return the wrong type, TypeScript catches it immediately.

06

Interfaces vs Type Aliases

1

📘 Definition

Both interface and type alias let you name and reuse a type shape. They look similar but have key differences that matter in real codebases.

2

📖 Official Insight

📖 TypeScript Docs say…

"The key distinction is that a type alias cannot be re-opened to add new properties vs an interface which is always extendable." — The official recommendation is: use interface for object shapes that may be extended (especially public APIs). Use type for unions, intersections, primitives, tuples, and computed types.

Featureinterfacetype alias
Object shapes✅✅
Declaration merging✅ Yes❌ No
Extends✅ extends keyword✅ intersection (&)
Union types❌ No✅ Yes
Mapped/conditional types❌ No✅ Yes
Primitives/tuples❌ No✅ Yes
4

⚙️ Syntax Comparison

TypeScriptinterface vs type
// Interface — extends with 'extends'
interface Animal {
  name: string;
  sound(): string;
}
interface Dog extends Animal {
  breed: string;
}

// Type alias — extends with intersection (&)
type Animal = { name: string; sound(): string };
type Dog = Animal & { breed: string };

// Declaration merging (only interface)
interface Window { myLib: unknown } // extends existing Window type

// Union — only possible with type alias
type ID = string | number;
type Result<T> = { data: T } | { error: string };
5

🔥 Real-World — React Component Props

TypeScript + ReactProps pattern
// Use 'type' for component props (common convention)
type ButtonProps = {
  label: string;
  variant: "primary" | "secondary" | "ghost";
  onClick: () => void;
  disabled?: boolean;
};

function Button({ label, variant, onClick, disabled }: ButtonProps) {
  return (
    <button
      className={`btn btn-${variant}`}
      onClick={onClick}
      disabled={disabled}
    >
      {label}
    </button>
  );
}

// Use 'interface' when you expect consumers to extend (library code)
interface BaseComponentProps {
  className?: string;
  style?: React.CSSProperties;
  children?: React.ReactNode;
}

interface CardProps extends BaseComponentProps {
  title: string;
}
6

⚠️ Pitfalls

❌ Declaration merging surprises

Interfaces can be declared multiple times — TypeScript merges them. This is powerful for augmenting third-party types but dangerous if you accidentally define two interfaces with the same name in your code — they silently merge instead of erroring.

Arrays & Tuples

1

📘 Definition

Arrays in TypeScript are typed collections of the same type. Tuples are fixed-length arrays where each position has a specific type — they model ordered, heterogeneous data.

4

⚙️ Syntax

TypeScriptArrays
// Array syntax — two equivalent forms
let names: string[] = ["Vipin", "Singh"];
let scores: Array<number> = [90, 85, 95];

// Readonly array — no mutation methods
const config: ReadonlyArray<string> = ["dev", "prod"];
// config.push("staging"); // ❌ readonly

// Array of objects
type User = { id: number; name: string };
const users: User[] = [
  { id: 1, name: "Vipin" },
  { id: 2, name: "Singh" },
];
TypeScriptTuples
// Tuple — fixed positions with fixed types
type Pair = [string, number];
const entry: Pair = ["Vipin", 25]; // ✅
// const bad: Pair = [25, "Vipin"]; // ❌ wrong order!

// Labeled tuple (more readable)
type NameAge = [name: string, age: number];

// Optional tuple element
type Response = [data: string, error?: string];

// Destructuring
const [userName, userAge] = entry;
// userName is string, userAge is number — fully typed
5

🔥 Real-World — React useState & Custom Hooks

TypeScript + ReactCustom hook with tuple return
// useState returns a tuple: [value, setter]
const [count, setCount] = useState(0); // [number, Dispatch<SetStateAction<number>>]

// Custom hook — return tuple for consistent destructuring
function useToggle(initial = false): [boolean, () => void] {
  const [state, setState] = useState(initial);
  const toggle = () => setState(s => !s);
  return [state, toggle]; // tuple — caller gets [boolean, fn]
}

const [isOpen, toggleOpen] = useToggle();
// isOpen: boolean, toggleOpen: () => void — fully typed
6

⚠️ Pitfalls

❌ Returning array from hooks without tuple annotation
TypeScriptPitfall
// Without tuple annotation, TS widens to (boolean | () => void)[]
function useToggle() {
  const [on, setOn] = useState(false);
  return [on, () => setOn(v => !v)]; // ❌ inferred as (boolean | (() => void))[]
}

// Fix 1: explicit return type
function useToggle(): [boolean, () => void] { ... }

// Fix 2: 'as const' on the return
return [on, () => setOn(v => !v)] as const;

Unions & Intersections

1

📘 Definition

A union type (A | B) represents a value that can be one of several types. An intersection type (A & B) represents a value that is all types simultaneously.

2

📖 Official Insight

📖 TypeScript Docs say…

Union: "A union type is a type formed from two or more other types, representing values that may be any one of those types." Intersection: "An intersection type combines multiple types into one. This allows you to add together existing types to get a single type that has all the features you need."

4

⚙️ Syntax & Patterns

TypeScriptUnion types
// Union of primitives
type ID = string | number;
let userId: ID = 1;
userId = "user_abc"; // ✅ both are valid

// Discriminated union — the most powerful pattern
type LoadingState = { status: "loading" };
type SuccessState = { status: "success"; data: string[] };
type ErrorState   = { status: "error";   message: string };

type AsyncState = LoadingState | SuccessState | ErrorState;

function render(state: AsyncState) {
  switch (state.status) {
    case "loading": return "Loading...";
    case "success": return state.data.join(", "); // TS knows data exists
    case "error":   return state.message;           // TS knows message exists
  }
}
TypeScriptIntersection types
// Intersection — combine multiple types
type Serializable = { serialize(): string };
type Loggable = { log(): void };

type Entity = Serializable & Loggable & { id: number };

// Entity must have ALL three: serialize(), log(), and id
const entity: Entity = {
  id: 1,
  serialize() { return `{"id":${this.id}}`; },
  log() { console.log(this); },
};
5

🔥 Real-World — Discriminated Unions in React

TypeScript + ReactAsync data state
type QueryState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: string };

function UserList({ state }: { state: QueryState<User[]> }) {
  if (state.status === "loading") return <Spinner />;
  if (state.status === "error")   return <Error msg={state.error} />;
  if (state.status === "success") return <List users={state.data} />;
  return null;
}
6

⚠️ Pitfalls

❌ Intersection of incompatible primitive types = never
TypeScriptPitfall
// A value can't be both string AND number at the same time
type Impossible = string & number; // resolves to: never
// Use intersections only for object types!

09

Introduction to Generics

1

📘 Definition

Generics are TypeScript's way of writing reusable, type-safe code that works with multiple types without losing type information. Instead of using any, generics let you parameterize types — like variables for types.

2

📖 Official Insight

📖 TypeScript Docs say…

"A major part of software engineering is building components that not only have well-defined and consistent APIs, but are also reusable. Components that are capable of working on the data of today as well as the data of tomorrow will give you the most flexible capabilities for building up large software systems." — Generics are how TypeScript achieves this.

3

🧠 Core Concept

TypeScriptThe problem generics solve
// WITHOUT generics — must duplicate or use any
function firstItem(arr: any[]): any { // ❌ loses type info
  return arr[0];
}

// WITH generics — T is a type parameter (placeholder)
function firstItem<T>(arr: T[]): T {
  return arr[0];
}

const num = firstItem([1, 2, 3]);          // num: number ✅
const str = firstItem(["a", "b"]);         // str: string ✅

// T is inferred from the argument — you don't need to pass it explicitly
4

⚙️ Syntax Patterns

TypeScriptGeneric functions, types, interfaces
// Generic function
function identity<T>(value: T): T {
  return value;
}

// Generic type alias

What is Generics

1

📘 Definition

Generics are TypeScript's mechanism for writing reusable, type-safe code that works across many types without sacrificing type information. Instead of accepting or returning any, a generic function or type accepts a type parameter — a placeholder that gets filled in at call time.

Think of generics as variables for types. Just as a function parameter holds a value at runtime, a type parameter holds a type at compile time.

2

📖 Official Insight

📖 TypeScript Docs say…

"A major part of software engineering is building components that not only have well-defined and consistent APIs, but are also reusable. A generic type gives us the ability to express that relationship [between input and output types]… We don't lose any information. We preserve the type of the input — we use a type variable, a special kind of variable that works on types rather than values."

3

🧠 Core Concept — The Problem Generics Solve

TypeScriptWithout generics — broken
// Option A: use any → loses all type safety
function identity(value: any): any {
  return value;
}
const result = identity("hello");
result.toFixed(2); // ❌ No TS error — runtime crash waiting to happen

// Option B: write a function per type → code duplication
function identityString(value: string): string { return value; }
function identityNumber(value: number): number { return value; }
// ❌ Doesn't scale — what about boolean, Date, User, Product...?
TypeScriptWith generics — type-safe and reusable
// T is a type parameter — a placeholder filled in at call time
function identity<T>(value: T): T {
  return value;
}

// TypeScript infers T from the argument
const s = identity("hello");   // T = string  → s: string
const n = identity(42);        // T = number  → n: number
const b = identity(true);      // T = boolean → b: boolean

s.toUpperCase(); // ✅ string method available
n.toFixed(2);   // ✅ number method available
🧠 Mental Model

A generic is like a template. You define the shape of logic once (<T>), and the caller fills in the actual type (string, User, etc.) at the point of use. TypeScript then checks that the filled-in type makes sense everywhere T is used in the function body.

Compared to Java/C# generics — same concept, but TypeScript's are structural. TypeScript checks what shape the type has, not what it's called.

4

⚙️ Syntax — Four Places Generics Appear

TypeScriptGenerics on functions, types, interfaces, classes
// 1. Generic function
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

// 2. Generic type alias
type Pair<A, B> = { first: A; second: B };
type StringNumberPair = Pair<string, number>; // { first: string; second: number }

// 3. Generic interface
interface Box<T> {
  value: T;
  unwrap(): T;
}

// 4. Generic class
class Stack<T> {
  private items: T[] = [];
  push(item: T): void  { this.items.push(item); }
  pop():  T | undefined { return this.items.pop(); }
  peek(): T | undefined { return this.items[this.items.length - 1]; }
}

const numStack = new Stack<number>();
numStack.push(1);
numStack.push("oops"); // ❌ Argument of type 'string' is not assignable to 'number'

Explicit vs Inferred Type Arguments

TypeScriptWhen to pass T explicitly
function wrap<T>(value: T): { data: T } {
  return { data: value };
}

// Inferred — TypeScript figures out T from the argument
const a = wrap("hello");      // T inferred as string

// Explicit — you specify T yourself (needed when TS can't infer it)
const b = wrap<string>("hello"); // same result, explicit

// You MUST pass T explicitly when there's no argument to infer from
async function fetchData<T>(url: string): Promise<T> {
  return fetch(url).then(r => r.json());
}
const user = await fetchData<User>("/api/user/1"); // must be explicit ✅

Advanced Generics

1

📘 Definition

Advanced generics extend the intro concept with constraints, default type parameters, multiple type parameters, and conditional/inferred types in generic position. They let you write APIs that are both flexible and narrowly type-safe.

2

📖 Official Insight

📖 TypeScript Docs say…

"You can declare a type parameter constraint using the extends keyword… Occasionally you'll want to set a default value for a type parameter as well." The docs distinguish between generic constraints (T extends U) and generic defaults (T = DefaultType).

3

🧠 Core Concepts

Constraints — T extends U

TypeScriptConstraining type parameters
// T must have at minimum { id: number }
function getById<T extends { id: number }>(items: T[], id: number): T | undefined {
  return items.find(i => i.id === id);
}

// T must be a key of object U  (keyof constraint)
function pluck<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { id: 1, name: "Vipin", role: "admin" };
pluck(user, "name");  // string ✅
pluck(user, "age");   // ❌ Argument of type '"age"' is not assignable

Multiple Type Parameters

TypeScriptK extends keyof T pattern
// Pick specific keys from an object — mirrors the Utility Type
function pick<T, K extends keyof T>(obj: T, keys: K[]): Pick<T, K> {
  const result = {} as Pick<T, K>;
  keys.forEach(key => { result[key] = obj[key]; });
  return result;
}
const partial = pick(user, ["id", "name"]);
// partial: { id: number; name: string } ✅

Default Type Parameters

TypeScriptGeneric defaults
// Default T to 'unknown' if not provided
type ApiResponse<T = unknown> = {
  data: T;
  status: number;
  error?: string;
};

type UserResponse = ApiResponse<User>;   // explicit
type AnyResponse  = ApiResponse;          // T defaults to unknown
4

⚙️ Syntax Patterns

TypeScriptConditional return type
// Return type narrows based on the input type
function coerce<T extends string | number>(val: T): T extends string ? number : string {
  return (typeof val === "string" ? Number(val) : String(val)) as any;
}

const a = coerce("42");   // number
const b = coerce(42);    // string

1

📘 Definition

TypeScript ships with a set of built-in generic types that perform common type transformations. These are called Utility Types. They save you from writing repetitive mapped types by hand and are the idiomatic way to derive new types from existing ones.

📖 Official Insight

"TypeScript provides several utility types to facilitate common type transformations. These utilities are available globally." Each utility type is implemented using mapped types and conditional types internally — you can inspect them in lib.es5.d.ts.

Type Narrowing & Control Flow Analysis

1

📘 Definition

Type narrowing is TypeScript's ability to refine a broad type into a more specific one within a conditional branch. TypeScript performs control flow analysis — it tracks which branch your code is in and updates its understanding of the type accordingly.

2

📖 Official Insight

📖 TypeScript Docs say…

"TypeScript uses control flow analysis to narrow types as you write conditional checks. Within an if block, TypeScript knows the type is more specific than before the check." The docs call this a form of type guard that TypeScript performs automatically — including typeof, instanceof, in, equality checks, and truthiness checks.

3

🧠 Core Concept — The Six Narrowing Techniques

TypeScriptAll narrowing forms
// 1. typeof narrowing
function format(val: string | number): string {
  if (typeof val === "string") return val.toUpperCase();  // val: string here
  return val.toFixed(2);                                  // val: number here
}

// 2. instanceof narrowing
function handleError(err: unknown) {
  if (err instanceof Error) return err.message; // err: Error
  return String(err);
}

// 3. 'in' narrowing — check if a property exists
type Cat = { meow(): void };
type Dog = { bark(): void };

function makeSound(animal: Cat | Dog) {
  if ("meow" in animal) animal.meow(); // animal: Cat
  else                  animal.bark(); // animal: Dog
}

// 4. Equality narrowing
function process(x: string | null) {
  if (x === null) return;  // x: null
  x.trim();               // x: string — null is excluded ✅
}

// 5. Truthiness narrowing
function greet(name: string | undefined) {
  if (!name) return "Hello, stranger";
  return `Hello, ${name}`; // name: string here
}

// 6. Discriminated union narrowing (most powerful)
type Shape =
  | { kind: "circle";  radius: number }
  | { kind: "rect";    width: number; height: number };

function area(s: Shape): number {
  switch (s.kind) {
    case "circle": return Math.PI * s.radius ** 2;
    case "rect":   return s.width * s.height;
  }
}
4

⚙️ Exhaustiveness Checking with never

TypeScriptnever as exhaustion guard
type Status = "pending" | "active" | "closed";

function getLabel(s: Status): string {
  switch (s) {
    case "pending": return "Pending";
    case "active":  return "Active";
    case "closed":  return "Closed";
    default: {
      // If you add "archived" to Status and forget this switch, TypeScript errors here
      const _exhaustive: never = s;
      throw new Error(`Unhandled status: ${_exhaustive}`);
    }
  }
}

Type Guards

1

📘 Definition

A type guard is a runtime check that narrows a type within a conditional block. TypeScript recognizes certain patterns as type guards automatically (typeof, instanceof, in). You can also write user-defined type guards using the is keyword to teach TypeScript about your custom checks.

2

📖 Official Insight

📖 TypeScript Docs say…

"A type predicate takes the form parameterName is Type. A predicate must refer to a parameter in the current function signature." Using is tells TypeScript: if this function returns true, then treat the argument as that type for the rest of the scope.

3

🧠 Core Concept

TypeScriptUser-defined type guards with 'is'
type AdminUser = { role: "admin"; permissions: string[] };
type RegularUser = { role: "user" };
type AnyUser = AdminUser | RegularUser;

// The 'is' return type makes this a type guard
function isAdmin(user: AnyUser): user is AdminUser {
  return user.role === "admin";
}

function showDashboard(user: AnyUser) {
  if (isAdmin(user)) {
    console.log(user.permissions); // user: AdminUser ✅
  } else {
    console.log("Restricted");      // user: RegularUser ✅
  }
}

Assertion Functions

TypeScriptasserts keyword — throw or narrow
// 'asserts' means: if this returns (doesn't throw), the type is narrowed
function assertIsString(val: unknown): asserts val is string {
  if (typeof val !== "string") throw new Error(`Expected string, got ${typeof val}`);
}

function processInput(val: unknown) {
  assertIsString(val);
  val.toUpperCase(); // val: string — TypeScript trusts the assertion ✅
}
4

⚙️ Patterns — Reusable Generic Guards

TypeScriptGeneric type guard factory

  obj: T, key: K

  return key in Object(obj);
}

// isDefined: filter out null/undefined from arrays
function isDefined<T>(val: T | null | undefined): val is T {
  return val !== null && val !== undefined;
}

const mixed: (string | null)[] = ["a", null, "b", null];

Function Overloading

1

📘 Definition

Function overloading lets you declare multiple type signatures for a single function. TypeScript uses these overload signatures for type-checking call sites, while a single implementation signature handles the actual logic.

2

📖 Official Insight

  | { ok: true;  value: T }

Conditional Types

1

📘 Definition

Conditional types let you express types that depend on a condition — they act like if/else logic but at the type level. The syntax mirrors JavaScript's ternary operator: T extends U ? X : Y. If T is assignable to U, the type resolves to X; otherwise Y.

They are the foundation of TypeScript's most powerful utility types like NonNullable, ReturnType, Parameters, and Awaited.

2

📖 Official Insight

📖 TypeScript Docs say…

"Conditional types take a form that looks a little like conditional expressions in JavaScript: SomeType extends OtherType ? TrueType : FalseType. When the type on the left of extends is assignable to the one on the right, you'll get the type in the first branch; otherwise you'll get the type in the latter branch." — The docs also call out distributive conditional types: when T is a naked type parameter and a union, the condition distributes over each member.

3

🧠 Core Concept

TypeScriptBasic conditional type
// "Is T a string? If yes → 'string-type', else → 'other-type'"
type IsString<T> = T extends string ? "string-type" : "other-type";

type A = IsString<string>;   // "string-type"
type B = IsString<number>;   // "other-type"

// Practical: unwrap a Promise — Awaited<T> is built on this
type Unwrap<T> = T extends Promise<infer U> ? U : T;

type C = Unwrap<Promise<string>>;  // string
type D = Unwrap<number>;            // number (not a Promise — passes through)

Distributive Conditional Types

TypeScriptDistributes over union members
// When T is a naked type parameter, the condition runs for EACH union member
type ToArray<T> = T extends unknown ? T[] : never;

type E = ToArray<string | number>;
// Distributes: (string extends unknown ? string[] : never) | (number extends unknown ? number[] : never)
// Result:      string[] | number[]

// NonNullable<T> from the standard library — removes null and undefined
type NonNullable<T> = T extends null | undefined ? never : T;

type F = NonNullable<string | null | undefined>; // string
4

⚙️ Syntax Patterns

TypeScriptNested conditionals + never filtering
// Extract only function members from an object type
type FunctionKeys<T> = {
  [K in keyof T]: T[K] extends Function ? K : never
}[keyof T];

type Obj = { id: number; name: string; save(): void; delete(): void };
type Methods = FunctionKeys<Obj>; // "save" | "delete"

// Deep readonly — conditional type recursion
type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K]
};
5

🔥 Real-World — Prisma-style select return type

TypeScriptConditional return based on select shape
type SelectResult<T, S extends Partial<Record<keyof T, boolean>>> = {
  [K in keyof T as S[K] extends true ? K : never]: T[K]
};

⚠️ Pitfalls

❌ Wrapping T in brackets kills distribution
TypeScriptNon-distributive form
// Distributive — T is naked
type Dist<T>    = T extends string ? "yes" : "no";
type R1 = Dist<string | number>; // "yes" | "no"  ← distributes

// Non-distributive — T is wrapped in []
type NoDist<T>  = [T] extends [string] ? "yes" : "no";
type R2 = NoDist<string | number>; // "no"  ← union not distributed

Use [T] extends [U] intentionally when you want to check the union as a whole — and avoid it accidentally when you do want distribution.

Mapped Types

1

📘 Definition

Mapped types let you create a new type by iterating over the keys of an existing type and transforming each property. They are the engine behind every built-in utility type: Partial, Required, Readonly, Pick, and Record are all mapped types internally.

The syntax is { [K in keyof T]: ... } — read as "for each key K in T, define a property".

2

📖 Official Insight

📖 TypeScript Docs say…

"A mapped type is a generic type which uses a union of PropertyKeys (frequently created via a keyof) to iterate through keys to create a type." The docs also describe mapping modifiers: you can add or remove readonly and ? optionality using + and - prefixes, and use key remapping via as to filter or rename keys.

3

🧠 Core Concept — Anatomy of a Mapped Type

TypeScriptHow the standard utility types work internally
// How Partial<T> is implemented in TypeScript's lib
type MyPartial<T> = {
  [K in keyof T]?: T[K]   // '?' adds optionality to every key
};

// How Required<T> removes optionality with '-?'
type MyRequired<T> = {
  [K in keyof T]-?: T[K]  // '-?' strips the optional modifier
};

// How Readonly<T> adds readonly with '+'
type MyReadonly<T> = {
  +readonly [K in keyof T]: T[K]
};

// How Mutable<T> removes readonly with '-readonly'
type Mutable<T> = {
  -readonly [K in keyof T]: T[K]
};
4

⚙️ Key Remapping with as

TypeScriptFilter, rename, and transform keys
// Filter out keys — map to 'never' to exclude them
type OmitFunctions<T> = {
  [K in keyof T as T[K] extends Function ? never : K]: T[K]
};

type User = { id: number; name: string; greet(): void };
type DataOnly = OmitFunctions<User>; // { id: number; name: string }

// Rename keys — prefix every key with "get"
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};

type UserGetters = Getters<{ name: string; age: number }>;
// { getName(): string; getAge(): number }

// Map values — transform every value type
type Nullable<T> = { [K in keyof T]: T[K] | null };
type Stringify<T> = { [K in keyof T]: string };

The infer Keyword

1

📘 Definition

infer is used inside a conditional type to declare a type variable that TypeScript will automatically infer from a matched position. Instead of you specifying the type, you ask TypeScript: "figure this out for me and give it a name I can use".

It is the mechanism behind ReturnType<T>, Parameters<T>, InstanceType<T>, and Awaited<T> in TypeScript's standard library.

2

📖 Official Insight

📖 TypeScript Docs say…

"Within the extends clause of a conditional type, it is now possible to have infer declarations that introduce a type variable to be inferred… infer placements represent types that TypeScript should extract from the matched type." The variable introduced by infer is only available in the true branch of the conditional.

3

🧠 Core Concept

TypeScriptinfer in action — how ReturnType works
// Built-in ReturnType<T> — implemented exactly like this
type MyReturnType<T extends (...args: any) => any> =
  T extends (...args: any) => infer R ? R : never;
  //                              ↑ TypeScript figures out R from the function

function getUser() { return { id: 1, name: "Vipin" }; }

type User = MyReturnType<typeof getUser>; // { id: number; name: string } ✅

// Built-in Parameters<T>
type MyParameters<T extends (...args: any) => any> =
  T extends (...args: infer P) => any ? P : never;

function createPost(title: string, body: string, userId: number) {}
type CreateArgs = MyParameters<typeof createPost>; // [title: string, body: string, userId: number]
4

⚙️ Patterns — infer in multiple positions

TypeScriptExtracting from arrays, promises, tuples
// Extract the element type from an array
type ElementOf<T> = T extends (infer E)[] ? E : never;
type A = ElementOf<string[]>;  // string
type B = ElementOf<User[]>;    // User

// Unwrap a Promise (same as built-in Awaited<T>)
type Awaited<T> = T extends Promise<infer U> ? Awaited<U> : T; // recursive!
type C = Awaited<Promise<Promise<string>>>; // string

// Extract first and last from a tuple
type Head<T extends any[]> = T extends [infer H, ...any[]] ? H : never;
type Tail<T extends any[]> = T extends [any, ...infer Rest] ? Rest : never;

type H = Head<[string, number, boolean]>; // string
type T = Tail<[string, number, boolean]>; // [number, boolean]
5

🔥 Real-World — Infer React component props

TypeScript + ReactExtract props from any component
import { ComponentProps, ComponentType } from 'react';

  T extends ComponentType<infer P> ? P : never;

⚠️ Pitfalls

❌ Using infer outside of a conditional type

infer is only valid inside the extends clause of a conditional type. Writing type X = infer T at the top level is a syntax error. If you find yourself wanting infer outside a conditional, you likely need ReturnType, Parameters, or a built-in utility instead.

Advanced Type Transformations

1

📘 Definition

Advanced type transformations combine conditional types, mapped types, infer, template literal types, and recursive types to derive complex new types from existing ones — entirely at compile time, with zero runtime cost. These are the techniques used inside TypeScript's own standard library and in libraries like Zod, tRPC, and Prisma.

2

📖 Official Insight

📖 TypeScript Docs say…

"Template literal types build on string literal types, and have the ability to expand into many strings via unions… When a union is used in the interpolated position, the type is the set of every possible string literal that could be represented by each union member." Combined with keyof, infer, and conditional types, this enables fully type-safe string manipulation at the type level.

3

🧠 Template Literal Types

TypeScriptString manipulation at the type level
// Template literal types — like JS template strings but for types
type EventName<T extends string> = `on${Capitalize<T>}`;

type A = EventName<"click">;   // "onClick"
type B = EventName<"focus">;   // "onFocus"

// Distribute over a union of strings
type DOMEvents = EventName<"click" | "focus" | "blur">;
// "onClick" | "onFocus" | "onBlur"

// Built-in string utility types
type U = Uppercase<"hello">;   // "HELLO"
type L = Lowercase<"WORLD">;   // "world"
type C = Capitalize<"foo">;    // "Foo"
type D = Uncapitalize<"Bar">;  // "bar"
4

⚙️ Recursive Types

TypeScriptJSON, deep path access, nested structures
// Recursive JSON type
type Json =
  | string | number | boolean | null
  | Json[]
  | { [key: string]: Json };

// Deep path access — get the type at a nested key path
type DeepGet<T, K extends string> =
  K extends `${infer Head}.${infer Tail}`
    ? Head extends keyof T ? DeepGet<T[Head], Tail> : never
    : K extends keyof T ? T[K] : never;

type Config = { db: { host: string; port: number }; app: { name: string } };
type H = DeepGet<Config, "db.host">; // string ✅
type P = DeepGet<Config, "db.port">; // number ✅

🔥 Real-World — tRPC-style type-safe router paths

TypeScriptTyped route key extraction
// Build a union of all dot-notation paths in a nested object
type DotPaths<T, Prefix extends string = ""> = {
  [K in keyof T & string]:
    T[K] extends object
      ? DotPaths<T[K], `${Prefix}${K}.`>
      : `${Prefix}${K}`
}[keyof T & string];

type AppRoutes = {
  users: { list: string; create: string };
  posts: { list: string; getById: string };
};
type Route = DotPaths<AppRoutes>;
// "users.list" | "users.create" | "posts.list" | "posts.getById" ✅

⚠️ Pitfalls

❌ Infinite recursion without a base case

Recursive types must have a base case that terminates the recursion. TypeScript has a recursion depth limit — hitting it gives a cryptic "Type instantiation is excessively deep" error. Always test recursive types with shallow inputs first, and add extends string ? ... : never style guards to short-circuit early.

Designing Reusable Type Systems

1

📘 Definition

A reusable type system is a set of generic, composable types that model your application's domain once — and then flow end-to-end from database schema to API response to UI component without duplication or casting. It is the pinnacle of TypeScript design: types that eliminate bugs by construction.

2

📖 Official Insight

📖 TypeScript Docs say…

"TypeScript's type system is structural — it cares about the shape of types, not their names. This means you can design type hierarchies that describe your domain without inheritance chains." Good type design leverages this: one well-shaped generic type can describe hundreds of specific cases.

3

🧠 Core Patterns

Pattern 1 — Brand Types (Nominal Typing)

TypeScriptPreventing primitive mix-ups
// TypeScript is structural — string is string, even if semantically different
function sendEmail(to: string, from: string) {}
sendEmail(userId, email); // ❌ args swapped — TS won't catch this

// Brand types: create a unique type identity for a primitive
type Brand<T, B> = T & { readonly __brand: B };

type UserId  = Brand<string, "UserId">;
type Email   = Brand<string, "Email">;

function createUserId(id: string): UserId  { return id as UserId; }
function createEmail(e: string):   Email   { return e  as Email; }

function sendEmail(to: Email, from: Email) {}

const uid = createUserId("u_123");
const em  = createEmail("v@example.com");

sendEmail(em, em);   // ✅
sendEmail(uid, em);  // ❌ Argument of type 'UserId' is not assignable to 'Email' ✅

Pattern 2 — Builder Pattern with Generics

TypeScriptType-safe query builder
// Each method call narrows the return type — chaining is type-safe
class QueryBuilder<T, Selected extends keyof T = keyof T> {
  private _fields: (keyof T)[] = [];

  select<K extends keyof T>(...fields: K[]): QueryBuilder<T, K> {
    this._fields = fields;
    return this as any;
  }

  build(): Promise<Pick<T, Selected>[]> {
    return fetch("/api/data").then(r => r.json());
  }
}

const result = await new QueryBuilder<User>()
  .select("id", "name")
  .build();
// result: Pick<User, "id" | "name">[] ✅

Pattern 3 — Discriminated Union State Machine

TypeScript + ReactAsync state — impossible states made impossible
// Model async state so idle/loading/success/error can't overlap
type AsyncState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error";   error: Error };

function UserCard({ state }: { state: AsyncState<User> }) {
  if (state.status === "loading") return <Spinner />;
  if (state.status === "error")   return <ErrorMsg msg={state.error.message} />;
  if (state.status === "success") return <div>{state.data.name}</div>; // state.data: User ✅
  return null;
}

⚙️ End-to-End Type Safety — Prisma → API → Client

TypeScript + Prisma + Next.jsTypes flow from DB schema to UI
// 1. Prisma generates these types from schema.prisma
import { User } from '@prisma/client';

// 2. Derive API shapes from the DB type — no duplication
type UserPublic    = Pick<User, "id" | "name" | "email">;
type CreateUserDto = Omit<User, "id" | "createdAt" | "updatedAt">;

// 3. Validate at the API boundary with Zod
const createUserSchema: z.ZodType<CreateUserDto> = z.object({
  name:     z.string(),
  email:    z.string().email(),
  password: z.string().min(8),
  role:     z.enum(["USER", "ADMIN"]),
});

// 4. Route handler — types from step 2 are already enforced
export async function POST(req: NextRequest): Promise<NextResponse<UserPublic>> {
  const parsed = createUserSchema.safeParse(await req.json());
  if (!parsed.success) return NextResponse.json(parsed.error as any, { status: 400 });
  const user = await db.user.create({ data: parsed.data });
  return NextResponse.json({ id: user.id, name: user.name, email: user.email });
}

⚠️ Pitfalls

❌ Manually duplicating types instead of deriving them

Writing a separate UserResponse type that manually re-lists every field from User is the #1 type maintenance problem in TypeScript apps. When the Prisma model changes, the manual copy gets out of sync. Always derive: Pick, Omit, Partial, and z.infer ensure your types stay in sync automatically.

Type Performance & Complexity

1

📘 Definition

Type performance is about how fast TypeScript's compiler resolves your types. Complex recursive types, deeply nested generics, and large union types can make the language server slow — causing poor editor experience, long CI builds, and "type instantiation is excessively deep" errors. Understanding what makes types slow lets you write fast, ergonomic type systems.

2

📖 Official Insight

📖 TypeScript Docs say…

"There are common ways to see whether your project has type-checking performance issues — running tsc --diagnostics will give you information about… instantiation counts and time spent on type checking." The TypeScript wiki also lists: prefer interfaces over type aliases for large object shapes (interfaces are cached; type aliases re-evaluated on each use), avoid excessive use of conditional types in hot paths, and use extends to constrain eagerly.

3

🧠 What Makes Types Slow

PatternCostAlternative
Deep recursive conditional types🔴 Very highAdd depth limit or use interface
Very large union types (100+ members)🔴 HighUse string + branding
type alias for large objects🟡 MediumUse interface (it's cached)
Re-computing the same mapped type repeatedly🟡 MediumAssign to a named type alias
Generic function with many overloads🟢 LowReduce overload count
4

⚙️ Practical Optimisation Techniques

TypeScriptinterface vs type for large shapes
// ❌ type alias — re-evaluated on every use in complex scenarios
type BigObjectType = {
  a: string; b: number; c: boolean; /* ... 50 more fields */
};

// ✅ interface — TypeScript caches the shape after the first check
interface BigObjectInterface {
  a: string; b: number; c: boolean; /* ... 50 more fields */
}
TypeScriptCaching expensive types + depth limiting recursion
// ❌ Inline computation repeated everywhere
function process(data: Omit<Pick<User, "name" | "email" | "role">, "role">) {}

// ✅ Name the derived type once — TypeScript reuses the cached result
type UserContact = Omit<Pick<User, "name" | "email" | "role">, "role">;
function process(data: UserContact) {}

// Depth-limited recursion — add a counter tuple to cap depth
type Prev = [never, 0, 1, 2, 3, 4, 5];

type DeepPartial<T, Depth extends number = 5> = [Depth] extends [0]
  ? T
  : { [K in keyof T]?: T[K] extends object ? DeepPartial<T[K], Prev[Depth]> : T[K] };

⚠️ Pitfalls

❌ Using 'as' to silence slow complex types

When a complex type is slow to resolve, the temptation is to cast with as to skip the check. This is wrong twice over: it hides the performance problem and removes type safety. Instead, simplify the type — name intermediate results, swap recursion for union enumeration, or use a simpler structural approximation.
