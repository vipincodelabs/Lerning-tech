01

What is TypeScript & How it Differs from JavaScript

1

📘 Definition

TypeScript is a strongly typed programming language that builds on JavaScript. It adds a static type system — meaning types are checked at compile time, before your code ever runs in a browser or Node.js.

TypeScript is a superset of JavaScript: every valid JavaScript file is also valid TypeScript. You add types on top; TypeScript compiles away to plain JavaScript.

FeatureJavaScriptTypeScript
TypingDynamic (at runtime)Static (at compile time)
Error detectionRuntime errorsCompile-time errors
Runs in browser✅ Directly❌ Must compile first
IDE supportLimited autocompleteFull IntelliSense + type hints
Refactoring safetyRiskySafe (types catch regressions)
File extension.js.ts / .tsx
2

📖 Official Insight

📖 TypeScript Docs say…

"TypeScript is a typed superset of JavaScript that compiles to plain JavaScript." — The TypeScript team describes it as a tool that "adds optional static typing" to JavaScript. The compiler is called tsc (TypeScript Compiler). The type annotations you write are erased at compile time — they have zero runtime cost.

3

🧠 Core Concept — The TypeScript Pipeline

Your code (.ts): const age: number = 25;
↓
TypeScript Compiler (tsc):
┌─────────────────────────────────────────┐
│ 1. Parse → AST (Abstract Syntax Tree) │
│ 2. Type-check → errors reported HERE │
│ 3. Emit → strip types, output .js │
└─────────────────────────────────────────┘
↓
Output (.js): const age = 25; ← types erased

This is the key insight: TypeScript types exist only during development. At runtime, JavaScript has no knowledge of them. TypeScript is purely a developer tool.

🧠 Mental Model

JavaScript is like writing code without spell-check. Errors show up only when you run it — often in production.

TypeScript is like writing with a smart editor that underlines mistakes before you even save the file. The spell-check disappears once the document is printed (compiled), but it saved you from errors.

4

⚙️ Syntax — Your First TypeScript

TypeScriptBasic
// JavaScript — no type annotation
let username = "Vipin";
username = 42; // ⚠️ No error in JS — silently wrong

// TypeScript — explicit type annotation
let username: string = "Vipin";
// username = 42; ❌ Type 'number' is not assignable to type 'string'

// Type annotation syntax: variableName: Type
let age: number = 25;
let isLoggedIn: boolean = true;
let greeting: string = "Hello";
5

🔥 Real-World Examples

TypeScriptWhy JS alone fails at scale
// JavaScript — no guardrails
function createUser(name, age, role) {
  return { name, age, role };
}

// What if a caller passes wrong args? JS has NO idea:
createUser(42, "admin", true); // silently accepted ❌

// ─────────────────────────────────────────────────
// TypeScript — contract enforced by the compiler
function createUser(name: string, age: number, role: string) {
  return { name, age, role };
}

createUser(42, "admin", true);
// ❌ Argument of type 'number' is not assignable to parameter of type 'string'
// Caught at compile time, not in production ✅
TypeScript + ReactProps typing
// Without TypeScript — any prop can be passed, no warnings
function UserCard({ name, age }) {
  return <div>{name} — {age}</div>;
}

// With TypeScript — caller knows exactly what to pass
type UserCardProps = {
  name: string;
  age: number;
};

function UserCard({ name, age }: UserCardProps) {
  return <div>{name} — {age}</div>;
}

// ❌ <UserCard name={42} age="twenty" />  — caught at compile time
// ✅ <UserCard name="Vipin" age={25} />   — correct
6

⚠️ Pitfalls

❌ Using `any` everywhere defeats TypeScript's purpose

any turns off type checking for that value. New developers often reach for any to silence errors — this gives you the worst of both worlds: no safety AND false confidence. Prefer unknown if you truly don't know the type.

❌ TypeScript ≠ runtime validation

TypeScript types are erased at runtime. If an API returns { age: "25" } instead of { age: 25 }, TypeScript won't catch it. Use Zod or io-ts for runtime validation of external data.

02

Type System Basics

1

📘 Definition

TypeScript's type system is the set of rules that determines which values can be assigned to which variables. It works by statically analyzing your code before it runs — this is called static analysis.

The type system has two modes: structural typing (the default, shape-based) and nominal typing (name-based, not default in TS). Understanding structural typing is fundamental to understanding why TypeScript behaves the way it does.

2

📖 Official Insight

📖 TypeScript Docs say…

TypeScript uses a structural type system. This means type compatibility is determined by the shape (structure) of the type, not its name. "If it walks like a duck and quacks like a duck, it's a duck." — This is often called duck typing in the TypeScript world.

3

🧠 Core Concept — Structural vs Nominal

TypeScriptStructural typing — shape is what matters
type Point2D = { x: number; y: number };
type Coordinates = { x: number; y: number };

const p: Point2D = { x: 1, y: 2 };
const c: Coordinates = p; // ✅ compatible — same shape!

// In a nominal system (Java/C#), this would fail.
// In TypeScript, if the SHAPE matches, it's compatible.

// Extra properties are OK when assigning (structural compatibility)
type Animal = { name: string };
type Dog = { name: string; breed: string };

const myDog: Dog = { name: "Rex", breed: "Husky" };
const animal: Animal = myDog; // ✅ Dog is assignable to Animal (has more, not less)
4

⚙️ Syntax — Type Annotations

TypeScriptAll annotation forms
// Variable annotation
let name: string;
const port: number = 3000;

// Function parameter + return type annotation
function greet(name: string): string {
  return `Hello, ${name}`;
}

// Object type annotation
let user: { name: string; age: number } = { name: "Vipin", age: 25 };

// Array annotation
let scores: number[] = [90, 85, 95];

// void — function that returns nothing
function logMessage(msg: string): void {
  console.log(msg);
}
5

🔥 Real-World — API Response Typing

TypeScriptTyped API response
// Define the shape of what your API returns
type ApiUser = {
  id: number;
  name: string;
  email: string;
  createdAt: string;
};

// TypeScript knows exactly what .name, .email are
async function fetchUser(id: number): Promise<ApiUser> {
  const res = await fetch(`/api/users/${id}`);
  return res.json() as ApiUser;
}

// Now the IDE gives you full autocomplete on the result
const user = await fetchUser(1);
console.log(user.name);  // ✅ TypeScript knows this is string
// console.log(user.xyz) // ❌ Property 'xyz' does not exist
6

⚠️ Pitfalls

❌ Excess property check at object literals — surprising behavior
TypeScriptPitfall
type User = { name: string };

// ❌ Fresh object literal triggers excess property check
const u: User = { name: "Vipin", age: 25 };
// Error: Object literal may only specify known properties

// ✅ Through a variable — no excess property check
const obj = { name: "Vipin", age: 25 };
const u2: User = obj; // fine — structural compatibility

TypeScript applies excess property checks only on fresh object literals assigned directly to a typed variable. This is intentional to catch typos.

03

Type Inference

1

📘 Definition

Type inference is TypeScript's ability to automatically determine the type of a value without you explicitly annotating it. You don't always need to write : string — TypeScript can figure it out from context.

2

📖 Official Insight

📖 TypeScript Docs say…

"In TypeScript, there are several places where type inference is used to provide type information when there is no explicit type annotation. For example, in this code: let x = 3; the type of x is inferred to be number." — TypeScript calls this contextual typing when inference flows from context (like event handlers).

3

🧠 Core Concept — How Inference Works

TypeScriptInference in action
// TypeScript infers 'string' from the initializer
let city = "Delhi";        // inferred: string
let count = 0;             // inferred: number
let active = true;        // inferred: boolean

// Inferred return type from function body
function add(a: number, b: number) {
  return a + b;            // return type inferred: number
}

// Array inference
const items = [1, 2, 3];  // inferred: number[]
const mixed = [1, "two"]; // inferred: (string | number)[]

// Contextual typing — TypeScript infers from context
const names = ["Vipin", "Singh"];
names.forEach(name => {
  console.log(name.toUpperCase()); // TS knows 'name' is string
});
4

⚙️ When to Annotate vs Infer

Rule of thumb:
• Let TypeScript infer when the type is obvious from the initial value.
• Annotate explicitly for: function parameters, function return types, variables declared without an initial value, and public API boundaries.

TypeScriptWhen to annotate
// ✅ Let inference work — obvious from initializer
const name = "Vipin";          // no annotation needed
const scores = [90, 85];       // inferred: number[]

// ✅ Annotate — declared without value
let userId: number;            // must annotate; no initializer
userId = 42;

// ✅ Annotate function parameters — TS can't infer these
function multiply(a: number, b: number): number {
  return a * b;
}

// ✅ Annotate return type — explicit contract, prevents mistakes
function getUsername(): string {
  return "Vipin";
}
5

🔥 Real-World — Inference in React useState

TypeScript + ReactuseState inference
import { useState } from 'react';

// TypeScript infers state type from initial value
const [count, setCount] = useState(0);     // count: number
const [name, setName] = useState("");      // name: string

// When type can't be inferred from initial value — annotate explicitly
const [user, setUser] = useState<User | null>(null);
// Without the generic, TS would infer: null — too narrow!
6

⚠️ Pitfalls

❌ Widened inference — `let` vs `const`
TypeScriptWidening
const status = "active";  // inferred: "active" (literal type)
let status2 = "active";  // inferred: string (widened!)

// 'const' infers the exact literal type because it can't change.
// 'let' widens to the broader type because it could change later.

Primitive & Special Types

1

📘 Definition

TypeScript has primitive types that map directly to JavaScript primitives, plus special types unique to TypeScript's type system that represent absence, unknowns, and impossibilities.

TypeWhat it representsExample
stringText"hello"
numberIntegers and floats42, 3.14
booleantrue/falsetrue
bigintArbitrary precision integers9007199254740991n
symbolUnique identifiersSymbol("id")
undefinedVariable declared, not assignedundefined
nullIntentional absence of valuenull
anyOpt-out of type checkingavoid!
unknownType-safe version of anyAPI responses
neverA value that never occursExhaustive checks
voidFunction returns nothing meaningfulEvent handlers
2

📖 Official Insight — any vs unknown vs never

📖 TypeScript Docs say…

any is the escape hatch — it disables type checking entirely. unknown is the type-safe counterpart: you can assign anything to it, but you must narrow it before using it. never represents values that never occur — it appears in exhaustive checks and functions that always throw.

3

🧠 Core Concept

TypeScriptany vs unknown vs never
// any — no type safety, avoid in production code
let data: any = fetchSomething();
data.doAnything();   // ✅ no errors — but unsafe!

// unknown — must narrow before using
let result: unknown = fetchSomething();
// result.doAnything(); // ❌ Object is of type 'unknown'

if (typeof result === "string") {
  console.log(result.toUpperCase()); // ✅ now safe
}

// never — exhaustive checking
type Shape = "circle" | "square";

function getArea(shape: Shape) {
  switch (shape) {
    case "circle":  return 3.14;
    case "square": return 1;
    default:
      const _exhaustive: never = shape; // TS error if new shape added!
      throw new Error(`Unknown shape`);
  }
}
4

⚙️ Literal Types

TypeScriptLiteral types
// A literal type restricts a variable to exact values
let direction: "left" | "right" | "up" | "down";
direction = "left";   // ✅
// direction = "diagonal"; // ❌

type HttpMethod = "GET" | "POST" | "PUT" | "DELETE";
type StatusCode = 200 | 201 | 400 | 401 | 404 | 500;

function request(url: string, method: HttpMethod) {
  // method is guaranteed to be one of the four values
}
5

🔥 Real-World — API Error Typing with unknown

TypeScriptError handling with unknown
// Before TypeScript 4.0, catch errors were 'any'
// Now they're 'unknown' with useUnknownInCatchVariables

async function fetchData(url: string) {
  try {
    const res = await fetch(url);
    return res.json();
  } catch (error: unknown) {
    // Must narrow before accessing properties
    if (error instanceof Error) {
      console.error(error.message);  // ✅ safe
    } else {
      console.error("Unknown error", error);
    }
  }
}
6

⚠️ Pitfalls

❌ Using `any` instead of `unknown` for external data

When you receive data from an API, catch block, or JSON.parse(), use unknown not any. With unknown you must validate before using — this forces safe code. With any you skip validation and runtime errors can occur.

05

Functions & Objects in TypeScript

1

📘 Definition

Functions in TypeScript can be typed at every boundary: parameters, return type, and even callback shapes. Objects get typed with inline type annotations, type aliases, or interface declarations.

4

⚙️ Syntax — Function Typing Patterns

TypeScriptAll function forms
// 1. Named function with parameter + return types
function greet(name: string): string {
  return `Hello, ${name}`;
}

// 2. Arrow function
const double = (n: number): number => n * 2;

// 3. Optional parameter (use ?)
function createUser(name: string, role?: string) {
  return { name, role: role ?? "user" };
}

// 4. Default parameter
function paginate(page: number = 1, limit: number = 10) {
  return { page, limit };
}

// 5. Rest parameters
function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}

// 6. Function type as a type alias
type Transformer<T> = (value: T) => T;
const double: Transformer<number> = n => n * 2;
TypeScriptObject typing
// Inline object type
function printUser(user: { name: string; age: number }) {
  console.log(user.name, user.age);
}

// Named type alias (preferred for reuse)
type User = {
  id: number;
  name: string;
  email: string;
  role: "admin" | "user";
};

// Readonly — prevents mutation
type Config = {
  readonly apiUrl: string;
  readonly timeout: number;
};

const config: Config = { apiUrl: "/api", timeout: 5000 };
// config.apiUrl = "other"; // ❌ Cannot assign to 'apiUrl' — it's readonly
5

🔥 Real-World — Express.js Typed Middleware

TypeScript + ExpressTyped middleware
import { Request, Response, NextFunction } from 'express';

// Extend Request to carry typed user data
interface AuthRequest extends Request {
  user?: { id: number; role: string };
}

function authMiddleware(
  req: AuthRequest,
  res: Response,
  next: NextFunction
): void {
  const token = req.headers.authorization;
  if (!token) {
    res.status(401).json({ error: "Unauthorized" });
    return;
  }
  req.user = { id: 1, role: "admin" }; // typed ✅
  next();
}
6

⚠️ Pitfalls

❌ Missing return type on exported functions

Always annotate return types on exported functions (API boundaries). Inference is fine for internal code, but an explicit return type acts as a contract — if you accidentally return the wrong type, TypeScript catches it immediately.

06

Interfaces vs Type Aliases

1

📘 Definition

Both interface and type alias let you name and reuse a type shape. They look similar but have key differences that matter in real codebases.

2

📖 Official Insight

📖 TypeScript Docs say…

"The key distinction is that a type alias cannot be re-opened to add new properties vs an interface which is always extendable." — The official recommendation is: use interface for object shapes that may be extended (especially public APIs). Use type for unions, intersections, primitives, tuples, and computed types.

Featureinterfacetype alias
Object shapes✅✅
Declaration merging✅ Yes❌ No
Extends✅ extends keyword✅ intersection (&)
Union types❌ No✅ Yes
Mapped/conditional types❌ No✅ Yes
Primitives/tuples❌ No✅ Yes
4

⚙️ Syntax Comparison

TypeScriptinterface vs type
// Interface — extends with 'extends'
interface Animal {
  name: string;
  sound(): string;
}
interface Dog extends Animal {
  breed: string;
}

// Type alias — extends with intersection (&)
type Animal = { name: string; sound(): string };
type Dog = Animal & { breed: string };

// Declaration merging (only interface)
interface Window { myLib: unknown } // extends existing Window type

// Union — only possible with type alias
type ID = string | number;
type Result<T> = { data: T } | { error: string };
5

🔥 Real-World — React Component Props

TypeScript + ReactProps pattern
// Use 'type' for component props (common convention)
type ButtonProps = {
  label: string;
  variant: "primary" | "secondary" | "ghost";
  onClick: () => void;
  disabled?: boolean;
};

function Button({ label, variant, onClick, disabled }: ButtonProps) {
  return (
    <button
      className={`btn btn-${variant}`}
      onClick={onClick}
      disabled={disabled}
    >
      {label}
    </button>
  );
}

// Use 'interface' when you expect consumers to extend (library code)
interface BaseComponentProps {
  className?: string;
  style?: React.CSSProperties;
  children?: React.ReactNode;
}

interface CardProps extends BaseComponentProps {
  title: string;
}
6

⚠️ Pitfalls

❌ Declaration merging surprises

Interfaces can be declared multiple times — TypeScript merges them. This is powerful for augmenting third-party types but dangerous if you accidentally define two interfaces with the same name in your code — they silently merge instead of erroring.

Arrays & Tuples

1

📘 Definition

Arrays in TypeScript are typed collections of the same type. Tuples are fixed-length arrays where each position has a specific type — they model ordered, heterogeneous data.

4

⚙️ Syntax

TypeScriptArrays
// Array syntax — two equivalent forms
let names: string[] = ["Vipin", "Singh"];
let scores: Array<number> = [90, 85, 95];

// Readonly array — no mutation methods
const config: ReadonlyArray<string> = ["dev", "prod"];
// config.push("staging"); // ❌ readonly

// Array of objects
type User = { id: number; name: string };
const users: User[] = [
  { id: 1, name: "Vipin" },
  { id: 2, name: "Singh" },
];
TypeScriptTuples
// Tuple — fixed positions with fixed types
type Pair = [string, number];
const entry: Pair = ["Vipin", 25]; // ✅
// const bad: Pair = [25, "Vipin"]; // ❌ wrong order!

// Labeled tuple (more readable)
type NameAge = [name: string, age: number];

// Optional tuple element
type Response = [data: string, error?: string];

// Destructuring
const [userName, userAge] = entry;
// userName is string, userAge is number — fully typed
5

🔥 Real-World — React useState & Custom Hooks

TypeScript + ReactCustom hook with tuple return
// useState returns a tuple: [value, setter]
const [count, setCount] = useState(0); // [number, Dispatch<SetStateAction<number>>]

// Custom hook — return tuple for consistent destructuring
function useToggle(initial = false): [boolean, () => void] {
  const [state, setState] = useState(initial);
  const toggle = () => setState(s => !s);
  return [state, toggle]; // tuple — caller gets [boolean, fn]
}

const [isOpen, toggleOpen] = useToggle();
// isOpen: boolean, toggleOpen: () => void — fully typed
6

⚠️ Pitfalls

❌ Returning array from hooks without tuple annotation
TypeScriptPitfall
// Without tuple annotation, TS widens to (boolean | () => void)[]
function useToggle() {
  const [on, setOn] = useState(false);
  return [on, () => setOn(v => !v)]; // ❌ inferred as (boolean | (() => void))[]
}

// Fix 1: explicit return type
function useToggle(): [boolean, () => void] { ... }

// Fix 2: 'as const' on the return
return [on, () => setOn(v => !v)] as const;
7

Unions & Intersections

1

📘 Definition

A union type (A | B) represents a value that can be one of several types. An intersection type (A & B) represents a value that is all types simultaneously.

2

📖 Official Insight

📖 TypeScript Docs say…

Union: "A union type is a type formed from two or more other types, representing values that may be any one of those types." Intersection: "An intersection type combines multiple types into one. This allows you to add together existing types to get a single type that has all the features you need."

4

⚙️ Syntax & Patterns

TypeScriptUnion types
// Union of primitives
type ID = string | number;
let userId: ID = 1;
userId = "user_abc"; // ✅ both are valid

// Discriminated union — the most powerful pattern
type LoadingState = { status: "loading" };
type SuccessState = { status: "success"; data: string[] };
type ErrorState   = { status: "error";   message: string };

type AsyncState = LoadingState | SuccessState | ErrorState;

function render(state: AsyncState) {
  switch (state.status) {
    case "loading": return "Loading...";
    case "success": return state.data.join(", "); // TS knows data exists
    case "error":   return state.message;           // TS knows message exists
  }
}
TypeScriptIntersection types
// Intersection — combine multiple types
type Serializable = { serialize(): string };
type Loggable = { log(): void };

type Entity = Serializable & Loggable & { id: number };

// Entity must have ALL three: serialize(), log(), and id
const entity: Entity = {
  id: 1,
  serialize() { return `{"id":${this.id}}`; },
  log() { console.log(this); },
};
5

🔥 Real-World — Discriminated Unions in React

TypeScript + ReactAsync data state
type QueryState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: string };

function UserList({ state }: { state: QueryState<User[]> }) {
  if (state.status === "loading") return <Spinner />;
  if (state.status === "error")   return <Error msg={state.error} />;
  if (state.status === "success") return <List users={state.data} />;
  return null;
}
6

⚠️ Pitfalls

❌ Intersection of incompatible primitive types = never
TypeScriptPitfall
// A value can't be both string AND number at the same time
type Impossible = string & number; // resolves to: never
// Use intersections only for object types!

09

Introduction to Generics

1

📘 Definition

Generics are TypeScript's way of writing reusable, type-safe code that works with multiple types without losing type information. Instead of using any, generics let you parameterize types — like variables for types.

2

📖 Official Insight

📖 TypeScript Docs say…

"A major part of software engineering is building components that not only have well-defined and consistent APIs, but are also reusable. Components that are capable of working on the data of today as well as the data of tomorrow will give you the most flexible capabilities for building up large software systems." — Generics are how TypeScript achieves this.

3

🧠 Core Concept

TypeScriptThe problem generics solve
// WITHOUT generics — must duplicate or use any
function firstItem(arr: any[]): any { // ❌ loses type info
  return arr[0];
}

// WITH generics — T is a type parameter (placeholder)
function firstItem<T>(arr: T[]): T {
  return arr[0];
}

const num = firstItem([1, 2, 3]);          // num: number ✅
const str = firstItem(["a", "b"]);         // str: string ✅

// T is inferred from the argument — you don't need to pass it explicitly
4

⚙️ Syntax Patterns

TypeScriptGeneric functions, types, interfaces
// Generic function
function identity<T>(value: T): T {
  return value;
}

// Generic type alias
type ApiResponse<T> = {
  data: T;
  status: number;
  message: string;
};

type UserResponse  = ApiResponse<User>;
type ProductList   = ApiResponse<Product[]>;

// Generic interface
interface Repository<T> {
  findById(id: number): Promise<T | null>;
  findAll(): Promise<T[]>;
  create(data: Partial<T>): Promise<T>;
  delete(id: number): Promise<void>;
}

// Generic with constraint — T must have an 'id' property
function findById<T extends { id: number }>(items: T[], id: number): T | undefined {
  return items.find(item => item.id === id);
}

🔥 Real-World — Typed API hooks with React Query

TypeScript + TanStack QueryGeneric API hook
import { useQuery } from '@tanstack/react-query';

// Generic reusable fetch function
async function apiFetch<T>(url: string): Promise<T> {
  const res = await fetch(url);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json() as T;
}

// Generic hook — works for any data type
function useApiData<T>(endpoint: string) {
  return useQuery<T>({
    queryKey: [endpoint],
    queryFn: () => apiFetch<T>(endpoint),
  });
}

// Usage — T is inferred from the explicit type parameter
const { data: users } = useApiData<User[]>("/api/users");
// users is User[] | undefined — fully typed ✅

⚠️ Pitfalls

❌ Using generics when a union type is better

Generics are for deferring the type — when the caller decides. If a function always accepts 2-3 specific types, use a union. Overusing generics makes code harder to read without real benefits.

Enums

1

📘 Definition

Enums (enumerations) let you define a set of named constants. Unlike union literal types, enums are one of TypeScript's few features that do generate JavaScript output — they exist at runtime.

2

📖 Official Insight

📖 TypeScript Docs say…

"Enums are one of the few features TypeScript has which is not a type-level extension of JavaScript." They compile to a JavaScript object. The docs also note: "In modern TypeScript, you may not need an enum when an object with as const could suffice."

4

⚙️ Syntax — Numeric, String, Const Enums

TypeScriptEnum variants
// 1. Numeric enum (default — values 0, 1, 2…)
enum Direction {
  Up,     // 0
  Down,   // 1
  Left,   // 2
  Right,  // 3
}
console.log(Direction.Up); // 0

// 2. String enum — preferred in practice (explicit, readable values)
enum UserRole {
  Admin  = "ADMIN",
  User   = "USER",
  Guest  = "GUEST",
}
console.log(UserRole.Admin); // "ADMIN"

// 3. Const enum — inlined at compile time (no runtime object)
const enum HttpStatus {
  OK      = 200,
  Created = 201,
  NotFound = 404,
}
// Compiled to: if (status === 200) — the value is inlined

// 4. Modern alternative — 'as const' object (no runtime overhead)
const Role = {
  Admin: "ADMIN",
  User:  "USER",
  Guest: "GUEST",
} as const;

type RoleType = typeof Role[keyof typeof Role]; // "ADMIN" | "USER" | "GUEST"
5

🔥 Real-World — Role-Based Access with Enums

TypeScript + ExpressRBAC with enums
enum UserRole {
  Admin  = "ADMIN",
  Editor = "EDITOR",
  Viewer = "VIEWER",
}

type User = {
  id: number;
  name: string;
  role: UserRole;
};

function canEdit(user: User): boolean {
  return [UserRole.Admin, UserRole.Editor].includes(user.role);
}

// Usage
const admin: User = { id: 1, name: "Vipin", role: UserRole.Admin };
canEdit(admin); // true
6

⚠️ Pitfalls

❌ Numeric enums allow any number to be assigned
TypeScriptNumeric enum unsafety
enum Status { Active = 1, Inactive = 2 }

function activate(s: Status) { ... }

activate(99); // ❌ No TS error! Numeric enums accept any number

// Fix: use string enums or as-const objects — they don't have this problem
enum Status { Active = "ACTIVE", Inactive = "INACTIVE" }
activate("anything"); // ❌ Correct error ✅
✅ When to use `as const` over enums

Prefer as const objects when: (1) you want zero runtime overhead, (2) you work with JSON/APIs where string values matter, (3) you don't need reverse mapping. Use enums when you need the enum's named access in runtime logic and your team finds them more readable.
